## 0. 章节定位与写作主张

### 0.1 本章要回答的问题

本章讨论的不是将一般地面 VLM 简单压缩后迁移到卫星，而是：在高分辨率遥感输入、有限 DRAM、受限功率与热控、辐射扰动和任务时限共同约束下，如何构建**可运行、可验证、可恢复**的单星 VLM 推理系统。

### 0.2 正文总论点

> 单星 VLM 部署的关键不是单点模型压缩，而是围绕视觉输入、模型结构、KV/运行时、硬件执行与可靠性保障，对“感知—推理—输出—恢复”全链路进行硬件感知协同重构。

### 0.3 全章统一约束表达

```text
M_peak ≤ M_available,
E_task ≤ E_available,
T_end-to-end ≤ T_deadline,
R_mission ≥ R_required.
```

| 符号 | 含义 | 正文中应报告的内容 |
| --- | --- | --- |
| `M_peak` | 推理峰值内存 | 权重、激活、KV cache、输入缓冲、运行时工作区 |
| `E_task` | 单任务能量消耗 | J/image、J/task、单轨能量预算与占空比 |
| `T_end-to-end` | 端到端任务时延 | prefill、decode、预处理、输出/下传等分项时延 |
| `R_mission` | 任务连续性/可用率 | 错误检测、恢复时间、回退成功率、热降频与中断行为 |

## 1. 问题定义：资源受限单星中的 VLM 推理闭环

### 1.1 从“模型上星”到“任务闭环”

#### 正文论点

星上部署的目标不是最大化离线吞吐，而是在给定任务窗口内稳定完成“输入处理—理解/生成—输出/下传—异常恢复”。视觉编码器、跨模态连接器和语言解码器分别引入视觉 token、模型权重与 KV cache 三类资源压力，因此模型可加载并不等价于可长期在轨运行。

#### 建议段落结构

1. 遥感输入具有高分辨率、多尺度、多光谱、多时相和视频流特征；
2. VLM 相比传统 CNN 额外引入视觉 token、语言模型权重和生成缓存；
3. 单星部署应同时闭合质量、内存、能量、时延、热控与可靠性；
4. 引出后续的全链路协同轻量化框架。

## 2. 面向星上资源约束的 VLM 执行链重构

> **本章核心节。** 技术按从输入到输出的执行路径组织，而非按量化、剪枝、蒸馏等算法名称平铺。

```text
星上原始观测
  → 任务感知预处理与 ROI 路由
  → 轻量视觉编码与 token 控制
  → 跨模态连接与紧凑语言解码
  → KV/内存感知运行时
  → 结构化输出、下传与故障回退
```

### 2.1 输入自适应与视觉 token 控制

#### 正文论点

遥感 VLM 的视觉 token 压缩不应追求均匀削减，而应形成“全局语境保留—局部高价值区域增强—冗余背景压缩”的非均匀资源分配机制，以兼顾星上资源预算和小目标、细长目标及空间关系的保真。

#### 技术机制

| 处理层 | 技术路线 | 目标 | 星上适配关注点 |
| --- | --- | --- | --- |
| 前置筛选 | 云/模糊过滤、轻量检测、变化检测、显著区域筛选 | 降低无效输入比例 | 前端模型低功耗、低时延、可回退 |
| 空间分解 | 全局缩略图 + ROI 高分辨率块 + 坐标/尺度编码 | 保留全局语义与局部细节 | 分块重叠、坐标编码、边界目标保护 |
| token 压缩 | patch merging、token merging、token pruning、窗口注意力 | 降低视觉序列长度与 prefill 开销 | 小目标召回、ROI 覆盖率、压缩前后 token 数 |
| 流式视觉编码 | 分块顺序编码、滑窗、关键帧/关键时相选择 | 避免一次性缓存全部输入 | 缓冲上限、块间关系、时相对齐 |

### 2.2 模型分部件轻量化与跨模态对齐保持

#### 正文论点

VLM 的视觉编码器、跨模态连接器、语言解码器和输出头对位宽、剪枝与蒸馏的敏感性不同；因此单星部署应采用分部件、混合精度和任务感知的紧凑化策略，而不是对整个模型施加统一低比特配置。

#### 技术机制

| 模型部件 | 主要约束 | 优先轻量化策略 | 关键风险 |
| --- | --- | --- | --- |
| 视觉编码器 | 高分辨率计算与激活 | 混合精度、结构化剪枝、蒸馏、层级编码 | 小目标与空间细节损失 |
| 跨模态连接器 | 投影开销与表征对齐 | 窄连接器、低秩投影、混合精度 | 视觉—语言对齐退化 |
| 语言解码器 | 权重、带宽与解码计算 | W8/W4、GQA/MQA、小型解码器 | 生成质量、事实一致性下降 |
| 输出头 | 固定字段或低复杂度输出 | 结构化头、模板化生成、任务蒸馏 | 开放式表达能力下降 |

#### 技术证据库

| --- | --- | --- | --- | --- |
| 权重量化 | PTQ/QAT、W8/W4、分层敏感性 | VLM quantization, LLM quantization | 峰值内存、时延、任务质量 | 待收集 |
| 激活/混合精度 | 高敏感模块的位宽保留策略 | mixed precision VLM, ViT quantization | 精度、kernel 支持、吞吐 | 待收集 |

### 2.3 KV cache、长序列与生成运行时管理

#### 正文论点

对包含问答、解释或报告生成的任务，KV cache 是决定单星生成式 VLM 能否稳定运行的关键内存变量。应从源头减量、缓存压缩、分页管理和上下文分层四个层面联合控制，而不是仅依赖权重量化。

#### 技术机制

| 层面 | 技术路线 | 目标 | 关键指标 |
| --- | --- | --- | --- |
| 源头减量 | GQA/MQA、限定输出长度、结构化输出 | 降低每 token 的 KV 占用 | KV head 数、最大生成长度 |
| 表示压缩 | KV 量化、关键 token 保留、历史 token 降精度 | 压缩缓存 | KV 精度、输出质量、事实一致性 |
| 内存管理 | 分页 KV、预分配块、缓存复用 | 降低碎片和重分配 | 峰值 KV、碎片、吞吐 |
| 上下文策略 | 系统指令/视觉 anchor/近期窗口/历史摘要分层 | 保留关键上下文 | 命中率、淘汰错误、上下文长度 |

## 3. 硬件感知编译、运行时与动态资源治理

### 3.1 从理论压缩率到物理加速比

#### 正文论点

模型 FLOPs、参数量或稀疏率下降并不会自动转化为星上时延与能耗下降。只有当量化位宽、剪枝结构、动态形状和内存布局被目标硬件内核与编译工具链有效支持时，算法压缩才会成为可测量的物理加速。

#### 技术机制

| 层级 | 要解决的问题 | 正文应报告的内容 |
| --- | --- | --- |
| 模型配置 | 位宽、剪枝结构、token 上限、GQA 配置 | W/A/KV 位宽、稀疏结构、最大 token 数 |
| 编译与图优化 | 算子融合、数据布局、内存复用、静态/动态形状 | 支持算子比例、回退算子、图优化策略 |
| 硬件执行 | 核利用率、带宽、温度、功耗模式 | 实测时延、能耗、温度、降频、吞吐 |

### 3.2 功率—热控约束下的动态执行

#### 正文论点

early exit、动态分辨率、动态 token 预算和模型回退应被统一理解为动态资源治理机制。运行时根据电量、温度、可用内存、任务时限、链路状态和模型置信度，在不同计算模式间切换。

| 模式 | 任务形态 | 执行链 | 资源目标 |
| --- | --- | --- | --- |
| M0 | 数据质量筛选 | 规则/轻量 CNN/小型视觉头 | 最低能耗、最大覆盖 |
| M1 | 视觉判别 | 轻量视觉编码器 + 任务头 | 快速分类、定位或变化检测 |
| M2 | 受限多模态输出 | 压缩 VLM + 结构化短输出 | 告警、属性归纳、固定字段报告 |
| M3 | 高价值复杂理解 | 完整或较完整 VLM 推理 | 复杂问答、解释和证据汇总 |

#### 技术证据库

| 技术点 | 需要整理的证据 | 检索方向 | 优先指标 | 状态 |
| --- | --- | --- | --- | --- |
| early exit | 退出策略、置信度校准和质量损失 | early exit transformer, anytime inference | 延迟、能耗、误判率 | 待收集 |
| 动态分辨率/token 预算 | 资源状态驱动的视觉复杂度调整 | dynamic resolution, adaptive token budget | token、能耗、小目标质量 | 待收集 |
| 任务调度 | 成像、推理、通信与充电的联合调度 | satellite AI scheduling, energy-aware inference | 任务收益、能量、时延 | 待收集 |
| 热感知降频与策略 | 温度状态对持续吞吐影响 | thermal-aware edge inference | 温度、降频、持续吞吐 | 待收集 |
| 模型回退 | 大小模型切换与服务连续性 | model fallback, cascaded inference | 可用率、恢复时延、质量 | 待收集 |

#### 写作边界

- 不将 early exit 仅写成网络结构技巧；
- 告警和安全关键任务应明确保守阈值与强制全推理条件。

### 3.3 推理期可靠性与降级运行

#### 正文论点

长期在轨运行中，可靠降级比理想环境下的峰值准确率更接近真实部署能力。本节只覆盖推理期的校验、恢复和回退；LoRA、Adapter、漂移检测与训练策略转入“持续适应：在轨微调”章节。

#### 技术机制与证据槽位

| 机制 | 作用 | 所需证据 | 状态 |
| --- | --- | --- | --- |
| 权重/模型哈希校验 | 防止权重包损坏或版本错配 | 星载软件、模型完整性或安全更新资料 | 待收集 |
| checkpoint 与恢复点 | 中断后续算或恢复任务状态 | 容错推理、星载计算资料 | 待收集 |
| A/B 版本与回滚 | 新模型或异常版本安全退出 | 软件部署/边缘系统资料 | 待收集 |
| 小模型/CPU 回退 | 主模型失败时维持最低任务能力 | 级联推理、容错系统资料 | 待收集 |
| 故障注入测试 | 覆盖 SEU/SEE、内存错误、热降频、链路中断 | 航天可靠性或在轨测试资料 | 待收集 |

## 4. 面向单星任务的部署配置与策略选择

### 正文论点

部署策略由任务价值、输入复杂度和可用资源共同决定，而不是由模型参数量单独决定。

| 任务/输入条件 | 建议模型形态 | 优先优化组合 | 不宜采用的策略 | 需要的证据 |
| --- | --- | --- | --- | --- |
| 高频数据质量筛选 | 轻量视觉模型或视觉编码器 + 小头 | INT8、固定输入、early exit、M0 | 长文本生成、复杂 KV 管理 | 在轨/边缘筛选案例 |
| 大幅面遥感理解 | 分块视觉编码器 + ROI 路由 + 短输出 VLM | 全局缩略图、ROI、token 合并、prefill 分段 | 所有原图 patch 平权送入 LLM | 高分辨率 VLM、遥感小目标证据 |
| 高光谱/多时相判别 | 流式编码 + 任务专用连接器 | 谱段选择、窗口缓存、分层加载 | 一次性拼接全部时相/谱段 | 高光谱/时序推理证据 |
| 高价值事件解释 | 小型 VLM 或分层调用 VLM | W4/W8、GQA、KV 量化、结构化输出、M2/M3 | 无限制开放生成 | VLM 端侧/长输出证据 |
| 资源/温度异常 | 轻量模型回退或延迟执行 | 动态 token 预算、降级模式、checkpoint | 维持满载长上下文推理 | 热感知/容错调度证据 |

### 写作边界

- 不在没有任务优先级假设的情况下宣称“某种模型规模最适合星上”；
- 不将 ISS 或大型卫星 HPC 的配置直接外推为 CubeSat 策略。

## 5. 单星部署的评测框架与报告规范

### 5.1 单星部署报告卡

每项技术或部署方案至少应报告以下五类结果：

| 维度 | 最低报告项 |
| --- | --- |
| 任务与输入 | 传感器、分辨率、波段/时相数、任务时效目标、输出长度 |
| 模型配置 | 参数量、视觉 token 上限、上下文长度、输出长度、量化位宽、GQA/KV 配置 |
| 资源效率 | 峰值 DRAM、存储占用、prefill/decode 时延、吞吐、J/image 或 J/task |
| 任务质量 | 任务精度、小目标召回、定位/边界误差、生成正确性、幻觉率、置信度校准 |
| 飞行可用性 | 热稳态吞吐、降频曲线、错误恢复时间、回退成功率、任务可用率 |

### 5.2 推荐对比基线

1. 未压缩或统一 FP16/BF16 模型；
2. 仅权重量化模型；
3. 仅视觉 token 压缩模型；
4. 全链路协同方案：量化 + token 控制 + KV 管理 + 运行时策略；
5. 资源异常或故障条件下的降级方案。

### 技术证据槽位

| 评价维度 | 需收集的现有 benchmark/案例 | 状态 |
| --- | --- | --- |
| VLM 端侧效率 | 峰值内存、prefill/decode、功耗 | 待收集 |
| 遥感任务质量 | 小目标、边界、多光谱、多时相指标 | 待收集 |
| 在轨资源与热控 | 任务功率、占空比、温度、降频 | 待收集 |
| 可靠性 | 错误、恢复、回退、可用率 | 待收集 |

## 6. 小结：从算法压缩到飞行级协同部署

### 建议正文收束段

> 单星部署的关键不在于寻找单一最优压缩算法，而在于以任务闭环为单位，将视觉 token 控制、模型分部件轻量化、KV/内存管理、硬件编译、动态功率—热控调度和可靠降级机制共同纳入部署设计。对资源受限卫星而言，真正有价值的方案应同时证明其任务质量、资源效率与长期运行可用性。

## 8. 调研执行计划与来源优先级

### 8.1 调研顺序

1. **第一优先级**：token 控制、量化、KV/长序列、流式 prefill、硬件感知运行时；
2. **第二优先级**：结构化剪枝、蒸馏、稀疏、MoE、early exit；
3. **第三优先级**：飞行级可靠性、故障恢复、动态功率/热控调度；
4. **最后补齐**：遥感特化、小目标、多光谱、多时相和实际在轨案例。

### 8.2 来源优先级

1. 原始论文的正式发表版本、作者公开版本或官方项目页；
2. 芯片、编译器、运行时和任务方的官方技术资料；
3. 真实在轨任务的一手论文、技术报告和任务方公告；
4. 高质量综述用于定位脉络，不作为关键数值或飞行结论的唯一依据；
5. 媒体、博客、二手转述只作为线索。

### 8.3 检索与核验原则

- 每篇关键论文至少核验：方法对象、实验平台、输入设置、量化/缓存配置、报告指标与局限；
- 每个“可上星”结论至少需要方法有效性、端侧/硬件执行和星上资源条件三层证据之一以上，并明确其证据缺口；
- 对性能数值保留精度口径、输入大小、批大小、上下文长度、功耗模式和测试硬件；
- 不把后续产品规格、地面开发板结果或同系列芯片辐照结果直接回填到已发射载荷。
