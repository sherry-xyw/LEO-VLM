## 一、 不同轨道的辐射环境分布

低地球轨道（LEO）受地球磁场保护，其辐射环境具有强烈的空间分布特征。算力星座的高度与倾角直接决定了其遭遇软错误的频次与类型。

| 轨道类型                   | 典型高度/区域                | 主要辐射来源                          | 辐射强度特点                  | 关键风险点                                                                                           |
| ---------------------- | ---------------------- | ------------------------------- | ----------------------- | ----------------------------------------------------------------------------------------------- |
| **LEO 低轨**（不含SAA、低倾角）  | 500~1500 km            | 内辐射带边缘质子（较弱）、GCR（被地磁部分屏蔽）、地球阴影区 | 总体强度最低，是辐射环境最温和的区段      | 持续暴露于GCR背景，强度随太阳活动周期波动                                                                          |
| **LEO + 南大西洋异常区（SAA）** | 同上，但经度40°W、纬度0°~30°S附近 | 内辐射带高能质子（>6.9 MeV）              | 范艾伦内带在此下陷至约200km，质子通量骤增 | 单次穿越剂量率可达平静区的1.5倍以上（STS-94实测吸收剂量率从约112升至约175 μGy/day）；质子通量沿纬度近似高斯分布，南向质子比北向多20%~30%，且日侧比夜侧强约20% |

| **辐射粒子类型**           | **能量与穿透力**                  | **核心物理机制**                            | **引起的软错误/硬错误类型**                                                                           |

## 四、 辐射错误对大模型的影响

### 1. 对模型参数（Weights）的影响

单粒子翻转（SEU）落在大模型参数里的影响,核心取决于**翻转的是哪一位、参数用什么格式存储、以及参数处在网络的什么位置**。

模型参数通常以浮点数(FP32/FP16/BF16)或量化整数(INT8)存储。以 FP32(IEEE 754)为例,32 位里包含 1 个符号位、8 个指数位、23 个尾数位,不同位翻转后果差异极大:
- **尾数位翻转**:数值只是小幅变动,对模型几乎无影响,通常被网络的冗余性吸收掉。
- **符号位翻转**:权重正负反转,影响中等。
- **指数位翻转(尤其最高位)**:这是最危险的情况。指数位决定数量级,翻转高位会让数值改变约 2^128 倍——一个 0.001 的权重可能瞬间变成 10³⁵ 量级,或者直接落入 Inf/NaN。
### 模型层面的实际后果
**最严重——推理崩溃或全盘失效**:一旦某个权重变成 NaN/Inf,它会随着矩阵乘法和激活在网络里向后传播,可能导致整层输出失效、最终结果全是 NaN,或推理直接报错。一个比特就能毁掉一次完整推理。
**中等——输出质量退化**:即使没产生 NaN,某个关键权重的大幅变化也会扭曲输出,表现为生成乱码、识别错误、置信度异常等,而且这种错误是"静默"的,系统不报错,很难当场发现。
**轻微——被冗余吸收**:大模型参数量巨大、存在大量冗余,对单个微小扰动有相当的韧性,因此随机翻转里的大多数其实不会带来可观察的影响。研究中常说的"关键位(critical bits)"现象就是指:少数位极其敏感,多数位无关紧要。

### 缓解手段
针对太空部署,通常多层防护并用:硬件上用 **ECC 内存**(可自动纠正单比特翻转)、辐射加固芯片、TMR 三模冗余;软件上做周期性权重校验和、冗余存储、看门狗检测;算法上对权重做**数值钳位**(限制范围,从根本上杜绝指数翻转导致的数值爆炸)、用量化或量化感知训练提升鲁棒性,以及对最敏感的关键参数做选择性保护。

### 2. 对 KV Cache（键值缓存）的影响

参数是静态常驻的,一个比特翻转会持续影响**之后每一次推理**,直到内存被纠正或权重重新加载。而 KV cache 是随请求生成、用完即弃的临时数据:
- 一次翻转**只污染当前这一条生成序列**,请求结束、缓存释放后,错误就随之消失。
- 下一个请求从全新的 cache 开始,具有"自愈"特性,不会累积。

如果系统启用了 **prefix caching / prompt caching**(把系统提示或公共前缀的 KV 状态缓存下来跨请求复用),那么击中这部分缓存的翻转就不再是瞬态的了——它会像参数翻转一样,**持续污染所有复用该前缀的后续请求**,直到缓存失效或被重算。设计辐射环境下的推理服务时,这是个容易被忽视的风险点。

KV Cache 的作用是帮大模型“记住”之前的对话和上下文，避免每生成一个新词都要重新计算前面的内容。

如果多模态大模型的 KV Cache 发生单粒子翻转，意味着**过去的记忆被篡改了**：

- **上下文丢失：** 模型在处理长文本或复杂指令时，会突然忘记前文的逻辑，导致前后矛盾。
    
- **幻觉激增：** 由于历史 Token 的向量表示变形，模型在“回看”之前的视觉或文本线索时会产生幻觉，无中生有地编造事实。

### 3. 对于tensor的计算和更新的影响

参数和 cache 都是"静态数据",翻转只能是数值错误。但计算过程同时涉及数据和控制,SEU/SET 可以击中两类完全不同的地方:

- **数据通路出错**:MAC/ALU/Tensor Core 算出一个错误的乘加结果,或累加器、中间寄存器里的值被翻转。后果是局部的、数值性的——某个输出元素错了。
- **控制通路出错**(参数和 cache 都不会有的全新风险):循环计数器、张量维度、地址/步长计算、指令译码被翻转。这类错误是**结构性**的,远比数值错误危险:一个被翻转的写地址会把结果写到错误的内存位置,**可能直接污染权重或 cache**(也就是说,计算错误能反过来"传染"成前两个问题里的持久错误),且完全静默、不可预测;一个被翻转的循环边界会导致迭代次数错误、输出结构错乱甚至挂死。

### 更新/训练:瞬态变持久的危险通道

"更新"二字把训练/在轨微调纳入了范围,而这正是计算环节最危险的地方——因为它提供了**让瞬态错误持久化**的路径:

- **梯度爆炸**:反向传播时一次计算被污染,产生一个巨大的错误梯度,可能让 loss 直接变 NaN,**一步之内毁掉整个模型**——所有权重被这个坏梯度更新成废值。这是单个比特造成全局灾难的最短路径。
- **优化器状态中毒**:Adam 的动量 m 和方差 v 是跨步骤累积的。一旦被污染,它会**影响之后许多步**的更新方向,污染像慢性毒药一样持续扩散,而不是一次性的。
- **权重写回**:被污染的更新值一旦写进权重,就**退化成第一个问题里的持久参数错误**——只不过这次是计算环节新引入的,之后每次推理都受影响。
- **无真值可比对**:训练过程中输出本就在变化,没有静态基准,这类错误极难当场发现

参数和 cache 都能靠 **ECC 内存**自动纠正单比特翻转,性价比很高;但 **ECC 只保护存储,不保护 ALU/Tensor Core 里正在进行的运算**,计算环节需要一整套不同的手段:

- **冗余执行**:双核锁步(lockstep)或 DMR/TMR,用两/三份计算结果比对,这是对抗瞬态计算错误的主力,而不是 ECC。
- **ABFT(基于算法的容错)**:针对 GEMM 量身定做——给矩阵加一行/一列校验和,矩阵乘法后验证就能检测甚至定位计算错误,开销远小于全量三倍冗余,是矩阵运算容错的经典方案。
- **数值钳位**:对中间结果和梯度做范围限制,从根上压制 Inf/NaN 爆炸。
- **训练专属防护**:**梯度裁剪**(直接挡住梯度爆炸)、**loss 突刺检测 + 回滚到检查点**、定期校验优化器状态。频繁 checkpointing 在辐射环境下的训练里几乎是必需品——一旦某步把权重算坏,能退回上一个干净状态。
- **控制逻辑加固**:针对那类最危险的地址/控制流错误,用辐射加固的控制单元、地址校验、看门狗,防止"写到错误位置污染权重"这种连锁灾难。

**一、并行策略**
**数据并行(Data Parallelism, DP)** 是最简单的:每张卡保存一份完整模型,把一个 batch 的数据切成若干份分给不同卡,各自算梯度,再通过 all-reduce 把梯度求平均、同步更新。它能线性提升吞吐,但有个致命问题——每张卡都要装下完整模型,**没有解决显存墙**。

**张量并行(Tensor Parallelism, TP),也叫层内并行**:把单个层内部的大矩阵运算横向切开。比如 `Y = XW`,把权重矩阵 W 按列切到不同卡上,每张卡算一块,再拼接/求和。它既减少了单卡的权重显存,又把每一层的计算并行化了。缺点是**通信极其频繁**(几乎每层都要通信),所以通常只在一台机器内部、用 NVLink 高带宽互连的几张卡之间做。这是 Megatron-LM 的招牌技术。

**流水线并行(Pipeline Parallelism, PP),也叫层间并行**:把模型按层切成几段(stage),每张卡负责连续的几层,数据像流水线一样逐段往后传。为了不让卡空等,会把 batch 切成多个 micro-batch 在流水线里"错峰"流动(GPipe、1F1B 调度)。它的通信量小(只在段边界传激活值),所以适合跨机器使用。主要挑战是**流水线气泡**——开头和结尾会有 GPU 空转。

**序列/上下文并行(Sequence/Context Parallelism, SP/CP)**:沿序列维度切分,专门对付超长上下文。当序列变长时,注意力和激活值的开销会爆炸式增长,这时把长序列切到多张卡上处理。

**专家并行(Expert Parallelism, EP)**:专门服务 MoE(混合专家)模型。把不同专家放到不同卡上,token 通过 all-to-all 被路由到对应专家。它能在不成比例增加计算量的前提下,把参数规模做得极大。

### PD 解耦
传统的推理方式是将一个请求的完整生命周期（算输入、吐字）全部在同一张显卡或同一个 GPU 集群上完成。而 PD 解耦则是**将“首字计算（Prefill）”和“后续生成（Decode）”物理隔离在不同的 GPU 机器或集群上**。

为了解决上述资源冲突，PD 解耦将集群划分为 **Prefill 集群** 和 **Decode 集群**：

#### 整个请求的处理流程如下：

1. **路由分发**：用户的请求（Prompt / 图像）首先到达全局调度器（Router），调度器将其指派给某一个 **Prefill 节点**。
    
2. **高速预填充（Prefill）**：Prefill 节点全速运转，利用高算力快速处理完输入，计算出该请求对应的 **KV Cache**，并吐出第一个字。
    
3. **KV Cache 跨网络传输（关键步）**：Prefill 节点将算好的巨额 KV Cache，通过数据通道（通常是高速的 RDMA 网络或 InfiniBand），远程发射（Offload）到调度器指定的 **Decode 节点**。
    
4. **持续自回归（Decode）**：Decode 节点接过现成的 KV Cache，开始专注、平稳地执行逐字生成。因为这个节点上**永远没有大 Prefill 请求来抢资源**，所以能保证极高的吐字吞吐量和极低的抖动。

### EPD 解耦
EPD 解耦（Encoder-Prefill-Decode Disaggregation）是多模态大模型（VLM / MLLM）在分布式推理服务（Serving）领域的最新前沿演进。它是对传统 **PD 解耦（Prefill-Decode）** 的进一步升级。

传统的 PD 解耦只把推理分为两个阶段（算输入和吐字）。但随着多模态大模型（如 Qwen2.5-VL、LLaVA）的发展，由于引入了高分辨率图像、长视频等高密度模态，整个推理执行路径天然地被拆解为了三个截然不同的阶段。

#### 1. Encoder 专用服务器（E 节点）

- **职责**：不加载任何语言模型（LLM）的权重。它只负责接收多模态输入（图片、视频），运行 ViT 等编码器将其转化为高维的**多模态嵌入（Embeddings）**。
    
- **硬件策略**：通常配置为 **TP=1（完全不切分张量）**，从而以最高效率压榨单卡算力。
    
- **核心外挂**：通常支持**全局多模态嵌入缓存（如结合 Mooncake 分布式存储）**。如果多个请求中包含重复的背景图片，E 节点直接从缓存提取 Embedding 发给下一步，连 ViT 编码都省了。
    
#### 2. Prefill 专用服务器（P 节点）

- **职责**：从 E 节点通过网络（如 gRPC 或 RDMA）接收处理好的多模态 Embedding，将其与用户的文本 Token 拼接，然后运行后端的 LLM 基座。
    
- **硬件策略**：专注于计算 $O(L^2)$ 复杂度的自注意力矩阵，初始化 **KV Cache** 并吐出第一个字。它属于传统的**计算密集型**，配置高 TP（如 TP=4 或 8）。
    
#### 3. Decode 专用服务器（D 节点）

- **职责**：通过超高速网络（如 InfiniBand / RoCE）接收从 P 节点远程“发射”过来的巨额多模态 KV Cache，然后开始专注于自回归地一个字一个字往下吐。
    
- **硬件策略**：属于典型的**显存带宽密集型**，通过 Continuous Batching（连续批处理）等技术让吐字吞吐量最大化、延迟最平滑。

### 数据传输信息压缩方法
#### 量化（对传输最友好）

- **核心观察**：key 有较大通道维异常值、value 更适合按 token 表示 → key 按通道、value 按 token 量化。
- 代表：**KIVI**（2-bit，免微调，per-channel key / per-token value）、**KVQuant**（面向千万级上下文，key 在 RoPE 前按通道量化）、**ZipCache**（显著 token 混合精度）、**WKVQuant**（两级方案）、**GEAR**（量化 + 低秩补偿误差）、**NVFP4 KV cache**（4-bit 存储、FP8 计算）。
- 极限：已有 **1-bit** KV cache 量化（Coupled Quantization、AsymKV 等）。
- **对传输的意义**：直接缩小要搬过 PD 边界的字节数，且不改序列结构 → **首选**。

####  token 淘汰 / 稀疏（缩短有效序列）

- **核心**：分析 token 重要性，丢弃次要的。代价是**不可逆信息损失**，对长程检索/精确召回不友好。
- 代表：**StreamingLLM**（attention sink + 近窗口）、**H2O**（累积注意力的 Heavy-Hitter）、**SnapKV**（观察窗口投票，针对 prefill）、**Scissorhands**（重要性持久假设）、**PyramidKV**（浅层多预算）、**HeadKV**（按头分配）。
- "不删而是动态加载"的稀疏注意力：**Quest**（按页选取）、**SparQ**（选重要通道）、**PQCache**（乘积量化检索 top-k）。

####  低秩 / 维度缩减

- **核心**：利用 KV 向量的隐藏维冗余，投影到低维。通常需额外训练或继续预训练。
- 代表：**MLA**（DeepSeek，低秩压缩 + 解耦 RoPE，已成事实基线）、**Palu**（投影矩阵低秩分解 + Fisher 信息秩搜索，兼容量化）、**Eigen Attention**（逐层秩分配）、**ShadowKV**（pre-RoPE key 低秩 + 解码时重建）、**CSKV**。

#### 跨层共享 / 深度压缩

- **核心**：相邻层 KV 相似 → 共享或合并。
- 代表：**GQA**（分组共享，最广泛采用的基线）、**KVsharer**（张量相似度跨层共享）、**MiniCache**（深度维度跨层压缩）、**XQuant**（跨层 + 超低比特）。

####  向量量化 / 字典编码

- 代表：**VQKV**（向量量化高保真高压缩比）、**CommVQ**（交换性向量量化）、**Lexico**（通用字典上的稀疏编码做极限压缩）。



### 大模型切分方法(核心)

从粗到细,目前可分为四类思路。SpaceMoE 给出了一个很清晰的演进谱系:**地面集中 → 星上集中(单星)→ 层级切分(分布式)→ 专家级切分**,后者可以把前几种看作特例。星上集中式推理把所有模型组件部署在单颗卫星上,只适用于中小规模模型;切分(分布式)推理按 Transformer 层划分,把不同层组分配给多颗卫星。 [arXiv](https://arxiv.org/html/2605.16849v1)

#### 1)层级切分 + 流水线并行(目前最主流的 LLM 方案)

代表作是 Zhang 等人 2026 年的 **Communication-Efficient Collaborative LLM Inference over LEO Satellite Networks**,这是目前对"星上大模型切分"最直接、最完整的一篇。它的几个设计正好对应上面的约束:

- **切分方式**:把整个 LLM 切成多个子模型,每个部署在一颗卫星上,通过星间交换中间激活来完成协同推理。 [arxiv](https://arxiv.org/pdf/2604.04654)
- **隐藏时延**:利用流水线并行机制,把子模型推理与中间激活传输重叠起来,从而降低推理时延。这是对"ISL 传输慢"的直接应对。 [arxiv](https://arxiv.org/pdf/2604.04654)
- **抑制误差累积**:设计了自适应激活压缩方案,在多级切分中缓解误差累积的同时保持推理精度。 [arxiv](https://arxiv.org/pdf/2604.04654)
- **联合优化建模**:把推理时延最小化问题建模为在有向无环图上的最短路径搜索——边权显式量化由切分和压缩策略带来的推理时延——并用改进的 A* 搜索算法求解。 [arxiv](https://arxiv.org/pdf/2604.04654)
- **效果**:与最先进基线相比,推理时延最多降低 42%、通信开销最多降低 71%,且推理精度损失小于 1%。 [arxiv](https://arxiv.org/pdf/2604.04654)

值得注意的是这里"切在哪一层"不是随意的。FourierCompress 等工作指出,现有动态切分策略(如 EdgeShard、Splitwise)主要基于系统资源,却忽略了不同层激活的内在数学特性——不同层激活的可压缩性差异巨大,选对切分点是设计高效压缩的前提。这意味着星间切分点的选择要同时考虑通信量、可压缩性和星上内存三者。 [arxiv](https://arxiv.org/pdf/2510.16418)

#### 2)专家级切分:SpaceMoE(2026,最具"星间原生"特色)

这是把 MoE 的稀疏性和卫星的"碎片化资源"对齐的思路,我认为是目前最有针对性的范式。SpaceMoE 是一种专家级切分推理架构,核心思想是把专家网络分布到多颗卫星上做并行计算。它有两个对星间环境很关键的优势:第一,它与 MoE 的内在稀疏性完全契合——由于专家激活通常高度倾斜(尤其在深层),源卫星可以本地缓存高频专家,从而大幅减少星间通信;第二,它支持每层内多星并行计算来加速推理,这与切分推理的串行计算形成鲜明对比。 [arXiv](https://arxiv.org/html/2605.16849v1)[arXiv](https://arxiv.org/html/2605.16849v1)

SpaceMoE 把问题拆成三个被星间约束"重塑"的子问题,这个分解框架很值得借鉴:

- **专家放置(长期)**:决定专家如何在星上有限内存下放置和复制。地面是静态优化,星上必须**移动性感知**——综合考虑卫星连通性的时间演化和星座规模,小星座需要更有选择性的放置,大星座允许更灵活的复制;由于激活分布倾斜,选择性地复制少量高频专家比均匀复制更有效;经常共激活的专家应尽量共置或就近放置。 [arXiv](https://arxiv.org/html/2605.16849v1)
- **专家选择(短期)**:决定每个 token 激活哪些专家。这里独特的是**电池退化感知**——当多个候选专家效用相近时,系统应倾向于选择能源条件更好、电池更健康的卫星上的专家,从而降低长期退化,在短期推理性能和长期系统可持续性之间取得平衡。 [arXiv](https://arxiv.org/html/2605.16849v1)
- **隐状态传输与路由**:在源卫星和专家卫星之间传中间隐状态。这里要做**轨道层级感知 + 重要性感知 + 热感知**三重路由——时延敏感的传输走层内路径,需要更稳定中继时走跨层路径;重要专家对应的隐状态走更可靠路径或低压缩,不重要的走低成本路径或激进压缩;不总是选最短路径,而是避开热裕量受限的卫星。 [arXiv](https://arxiv.org/html/2605.16849v1)

它提炼的三组核心权衡可以直接当作星间切分的设计准则:访问时延 vs 存储效率(专家放置);短期推理性能 vs 长期电池可持续性(专家选择);推理精度 vs 散热效率(隐状态传输与路由)。
