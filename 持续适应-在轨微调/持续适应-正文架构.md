# 持续适应：正文架构

> **章节定位**：本章讨论视觉—语言模型在低轨单星与星座环境中，如何在观测分布、传感器、任务指令和外部知识持续变化时，以受控的资源代价完成知识更新、参数适配、协同同步与安全发布。
>
> **核心论点**：在轨持续适应不应被理解为周期性重训，而应被建模为由变化触发、分级更新、局部验证、异步协同与可回滚发布组成的闭环。模型参数更新只是其中一个环节；对于不同变化，应优先选择代价更低、风险更可控的检索记忆、Prompt、Adapter、LoRA 或局部任务头更新。

---

## 1. 问题起点：为什么单星与星座 VLM 必须持续适应

### 1.1 非平稳观测环境使离线模型逐步失配

遥感 VLM 的部署环境同时存在多类变化：

- **观测分布变化**：地域、季节、太阳高度、云雾、海况、灾害阶段与成像几何变化；
- **传感器与模态变化**：可见光、SAR、红外、多光谱、高光谱及其分辨率、谱段和噪声特性变化；
- **任务变化**：新增目标类别、告警规则、问答任务、描述粒度和输出格式变化；
- **知识变化**：区域背景、目标案例、事件报告、任务先验和外部事实持续更新；
- **系统变化**：能量、热状态、存储、链路窗口和可用算力变化，使同一更新方案在不同轨道时段具有不同可行性。

CLeaRS 将遥感 VLM 的连续变化归纳为长时程、模态增量和任务增量三类协议，并显示现有模型在这些条件下普遍出现灾难性遗忘。[R1]

### 1.2 适应目标不是“始终更新”，而是“在必要时以最小风险更新”

持续适应的目标应定义为：在满足任务时效、能耗、链路和安全约束的前提下，使模型对当前任务保持足够可靠，同时尽可能维持既有任务能力。可用如下决策表达式概括：

```text
max   ΔU_new − λ_f·F_old − λ_r·C_update
s.t.  M_train ≤ M_available,
      E_update ≤ E_available,
      T_update ≤ T_window,
      R_release ≥ R_required.
```

其中，`ΔU_new` 表示新任务或新环境下的效用提升，`F_old` 表示对既有任务能力的遗忘，`C_update` 表示训练、通信和验证代价；约束分别对应训练内存、能量、可用窗口与发布可靠性。

### 1.3 本章与相邻章节的边界

- **单星部署**关注模型在给定版本下如何高效、稳定地完成推理；
- **持续适应**关注模型、知识库和策略如何在运行期间发生受控更新；
- **模型切分与协同推理**关注模型组件在星间或星地如何分配执行；
- 本章只在训练能力不足时讨论 split-PEFT 作为更新路径的备选，不重复展开通用切分推理。

## 2. 变化识别与更新触发：先判断“是否需要更新”

### 2.1 变化检测应覆盖数据、任务、知识与系统四个层面

持续适应不能以固定周期盲目执行。更新触发应由多源信号联合决定：

| 变化层面 | 可观测信号 | 典型风险 | 建议的初始响应 |
| --- | --- | --- | --- |
| 数据分布 | 特征距离、预测熵、置信度校准误差、OOD 分数 | 视觉表征失配 | 增加观测、触发自监督预适应或 PEFT 候选 |
| 模态/载荷 | 谱段缺失、新传感器接入、分辨率变化 | 跨传感器迁移失败 | 模态适配器或连接器更新 |
| 任务与指令 | 新类别、规则改变、输出字段变化 | 任务语义失配 | Prompt / Prefix / 任务头更新 |
| 知识与案例 | 检索召回下降、证据冲突、案例覆盖不足 | 幻觉、错误解释 | RAG 索引、案例库或摘要更新 |
| 系统资源 | 电量、温度、DRAM、链路、窗口长度 | 更新中断或影响主任务 | 延迟、降级、分批更新或仅更新记忆层 |

### 2.2 分级触发机制

建议采用从低风险到高成本的四级更新路径：

```text
T0：不更新参数，仅维持监测与常规推理
T1：更新检索库、案例记忆、任务配置或规则
T2：更新 Prompt、Adapter、LoRA 或局部任务头
T3：触发多星协同聚合、地面复核或受控版本发布
```

其核心原则是：**知识变化优先采用非参数化更新；短期任务变化优先采用 Prompt；持续域失配或新模态接入再触发 PEFT；跨星共享仅在局部验证通过后进行。**

FedDAA 对动态概念漂移的区分与客户端分群，可用于构建“单星检测—轨道级分群—分组更新”的星座触发逻辑；但其证据来自通用联邦学习，需要在遥感时空数据上重新定义漂移信号与阈值。[E4]

### 2.3 本节正文应形成的结论

> 在轨持续适应的第一步不是微调，而是确定变化的性质与更新必要性。将所有性能波动直接归因于参数失配，会导致不必要的训练、通信和遗忘风险；更合理的方式是根据数据、任务、知识和资源状态，对更新对象与更新粒度进行分级选择。

## 3. 分级更新对象：从非参数知识到参数高效适配

### 3.1 非参数化更新：RAG、案例记忆与任务知识库

当变化主要来自区域知识、历史案例、任务规则或外部事实时，优先更新检索库、向量索引、证据摘要与任务缓存，而非直接修改模型权重。

RS-RAG 将遥感图像与文本知识编码到统一向量空间，通过检索、重排序和知识增强提示支持图像描述、分类与 VQA；SAR-RAG 则将具有已知属性的历史目标样本作为 SAR 识别和解释的检索记忆。[R6][R7]

对星上系统，可将记忆按**地域—时间—载荷—任务—版本**建立命名空间，并维护：

- 地理对象、目标样本和事件证据；
- 传感器元数据与成像质量记录；
- 历史告警、人工确认与地面反馈；
- 检索可信度、证据冲突与过期状态。

本节重点不在于介绍一般 RAG，而在于回答：什么知识可以以索引或案例形式更新、何时检索失效需要升级到参数更新、以及多星记忆如何去重和保持来源可追溯。

### 3.2 Prompt / Prefix 更新：面向任务语义和输出约束的轻量适配

Prompt、Prefix 或任务模板适合处理任务说明、输出字段、告警等级、问答侧重点和临时规则变化。它们不应与运行时的自然语言重设任务混同：前者是可学习或可优化的提示参数，后者是用户输入或策略配置。

TPPT 以稳定文本原型引导视觉或视觉—文本 Prompt 的增量学习，为“优先尝试低成本 Prompt 更新，并用稳定语义锚点控制表征漂移”提供可借鉴机制。[E3]

### 3.3 Adapter / LoRA 更新：面向持续域失配、新模态与复杂任务变化

当检索和 Prompt 不能解决持续的观测域失配、传感器变化或任务能力缺失时，应选择可控的参数高效更新。

- **Adapter**适合模块化任务隔离、按需路由和多任务并存；
- **LoRA**适合在冻结基座上注入低秩更新，降低可训练参数和上行更新包大小；
- **局部任务头**适合目标检测、分类、变化检测等判别任务的快速适配；
- **连接器/投影层**适合新视觉编码器、新谱段或新模态引入时的跨模态对齐。

LoRA-Det 表明，在星载遥感有向目标检测的研究设定中，仅更新约 12.4% 的参数即可获得接近全量微调的性能；Geospatial FM PEFT 与 SpectralGPT PEFT 则说明参数高效方法可用于地理空间与光谱基础模型的跨任务、跨区域或跨谱段适配。[R3][R4][R5]

### 3.4 更新对象选择规则

| 变化类型 | 首选更新对象 | 升级条件 | 不宜优先采用的策略 |
| --- | --- | --- | --- |
| 新知识、案例、区域背景 | RAG / 向量库 / 证据摘要 | 检索失效持续、事实冲突无法消解 | 直接全参数微调 |
| 输出格式、任务描述、临时规则 | Prompt / Prefix / 策略文件 | Prompt 无法恢复任务性能 | 直接修改视觉编码器 |
| 持续地域或季节失配 | Adapter / LoRA | 新旧能力差异持续扩大 | 固定规则或仅靠检索 |
| 新传感器、新谱段或模态缺失 | 连接器、模态 Adapter、低秩模块 | 跨模态对齐失败 | 使用统一更新 rank 覆盖所有模块 |
| 新类别或判别任务 | 局部任务头或检测器 LoRA | 任务与语义结构显著变化 | 开放式生成替代结构化头 |

### 3.5 本节正文应形成的结论

> 持续适应应采用“知识优先、参数按需”的层级更新策略。检索记忆、Prompt、Adapter、LoRA 和任务头并非彼此替代，而是对应不同变化来源、更新成本和遗忘风险的互补工具。

## 4. 单星局部适配：数据、训练、遗忘与资源约束的联合设计

### 4.1 更新数据的来源与可信度

单星在线更新不应默认依赖大规模人工标注。可用监督信号可分为：

- 地面站回传的高可信标签与任务反馈；
- 与多源传感器、轨道位置或时序一致性结合的弱监督信号；
- 置信度门控后的伪标签；
- 无标注观测流上的自监督表征更新；
- 人工审核后的高价值错误案例。

CoSMAE 提供了持续自监督遥感表征更新的思路：通过数据混合与模型混合蒸馏维持当前和历史分布之间的表征联系。它可作为“标签不足时先适应视觉表征、获得可靠监督后再进入任务级 PEFT”的候选路径。[R2]

### 4.2 遗忘控制应与更新算法共同设计

对新观测的局部适配必须同时保护既有任务能力。建议将以下机制作为一个联合模块讨论：

- **回放与代表样本选择**：保留跨地域、跨季节、跨传感器和高风险任务的少量代表样本；
- **蒸馏与功能保持**：以旧模型、旧 Adapter 或历史输出约束更新后的行为；
- **参数隔离**：以 Prompt、Adapter pool 或 LoRA bank 隔离不同任务与模态更新；
- **模块路由**：依据载荷、区域、任务或漂移群组选择特定适配器；
- **不确定性门控**：低置信更新不得替代稳定版本。

CLeaRS 的遗忘结果表明，仅对顺序任务进行 LoRA 微调不足以解决任务、指令与模态切换问题；因此，PEFT 不是遗忘控制本身，仍需与记忆、蒸馏、路由或联合验证结合。[R1]

### 4.3 单星更新的资源门控

单星局部训练至少需要满足：

```text
M_train ≤ M_available,
E_update + E_inference_reserve ≤ E_available,
T_train + T_validate ≤ T_window,
Q_new − F_old ≥ τ_release.
```

其中，`M_train` 包括可训练参数、优化器状态、梯度、激活与回放缓存；`E_inference_reserve` 用于保证主任务推理不被更新任务挤占；`Q_new` 与 `F_old` 分别表示新任务收益与旧任务遗忘；`τ_release` 是版本发布阈值。

当完整 PEFT 超出单星预算时，卫星—地面协同微调可以作为受限备选，而不是把模型切分默认为更优方案。Satellite Federated Fine-Tuning 将冻结嵌入层和可训练任务头置于卫星、将冻结骨干置于地面，并在轨内聚合任务头；其仿真中训练时间降至完全板上训练的约 33%。但该收益以传输嵌入/特征向量为代价，且只更新任务头。因此，部署前必须以实际模型比较 `embedding/feature bytes`、`activation/gradient bytes` 与 `adapter delta bytes`，并计入窗口完成率和隐私边界。[R8]

### 4.4 本节正文应形成的结论

> 单星参数高效适配并不等于低成本、低风险适配。除可训练参数比例外，还必须同时报告监督来源、峰值训练内存、能耗、训练窗口、旧任务遗忘和更新后不确定性，才能判断其是否适合在轨执行。

## 5. 局部验证与安全发布：将“可训练”转化为“可部署”

### 5.1 更新前后的联合验证

每一次更新应至少比较：

1. 新环境或新任务上的收益；
2. 历史关键任务上的性能保持；
3. 检索证据一致性与生成可靠性；
4. 训练、推理与通信资源是否仍满足约束；
5. 更新包的完整性、来源和版本兼容性。

建议采用“影子推理—小比例灰度—稳定版本切换”的发布过程：

```text
候选更新
  → 本地影子评测
  → 新旧任务联合门控
  → 小比例任务灰度运行
  → 持续监测
  → 正式发布或自动回滚
```

### 5.2 版本化对象必须独立管理

需要独立编号、签名、验证与回滚的对象包括：

| 对象 | 典型更新原因 | 回滚依据 |
| --- | --- | --- |
| 基座模型 | 重大架构或安全更新 | 兼容性失败、性能退化 |
| Prompt / 策略文件 | 任务语义和输出规则变化 | 任务一致性、格式错误 |
| Adapter / LoRA | 域、模态或任务适配 | 忘记率、置信度、资源超限 |
| 检索索引 / 记忆 | 新案例、新知识、过期证据清理 | 检索 Recall、冲突率、时效性 |
| 聚合状态 | 星座协同更新 | 陈旧度、版本分叉、群组异常 |

### 5.3 本节正文应形成的结论

> 在轨微调的安全性不由“是否使用 LoRA”决定，而由更新前验证、版本一致性、灰度发布、异常监测和回滚能力共同决定。只有将这些机制纳入运行时，参数更新才具有飞行可用性。

## 6. 星座协同适应：从单星更新到异步聚合与多星记忆合并

### 6.1 为什么不能假设全星同步

低轨星座存在可见窗口错位、节点计算差异、能源差异、链路中断和时空非 IID 数据。因而，多星协同不应使用“所有卫星同时训练、统一 FedAvg”的理想化假设。

AsyncFLEO 通过卫星分组、模型传播与陈旧度折扣处理窗口错位和更新过期；FEDMEGA 采用轨道内 ring all-reduce 与全局流量调度，为轨道内先聚合、星地再协调提供系统骨架。[R11][R10]

On-board Federated Learning 进一步说明，卫星可预测的轨道访问和同轨 ISL 可以用于模型分发与逐跳增量聚合；但它把更新视为一般模型向量，尚未解决 VLM/LoRA 的异构 rank、任务冲突和版本语义问题。[R9]

### 6.2 推荐的三级协同结构

```text
单星：本地记忆更新、PEFT、局部验证
  → 轨道内：按载荷/区域/任务/资源分组，异步聚合更新差分
  → 星地：全局验证、跨轨道协调、签名发布与稳定版本维护
```

其中，轨道内聚合的通信对象不应默认是完整模型，可包括：

- LoRA/Adapter delta；
- 参数化知识 adapter；
- 索引差分、案例摘要与冲突标记；
- 置信度统计、漂移摘要和版本元数据。

### 6.3 异构 rank、模态与任务下的联邦 LoRA

不同卫星可能使用不同 LoRA rank、不同可训练层、不同传感器或不同任务集。此时，直接平均 LoRA 因子既可能受到维度不匹配影响，也可能因低秩分解的表示不唯一而改变实际更新语义。

FediLoRA、ILoRA 与 GLoRA 分别从维度级聚合、异构 rank 对齐以及服务器端共识更新子空间出发处理这一问题；GLoRA 特别指出，联邦 LoRA 的聚合对象应是语义上可对齐的更新表示，而非 A/B 因子的朴素均值。[R14][R15][R16]

### 6.4 多星记忆合并

星座中的记忆更新不应直接合并全部案例或检索索引。需要：

- 以时空位置、载荷、任务与版本区分记忆命名空间；
- 识别重复、冲突、过期和低可信证据；
- 将原始图像/文本与可共享摘要、参数化知识适配器分离；
- 基于相关性和冲突约束选择性合并，而非无差别平均。

FedMosaic 以参数化 adapter 表示知识、用文档专属 mask 与选择性聚合缓解冲突知识合并，为多星知识共享提供可参考的抽象机制；但其仍是地面语言 RAG 证据，遥感时空关系与多传感器案例需进一步验证。[E2]

### 6.5 本节正文应形成的结论

> 星座协同持续适应的关键不是将单星 LoRA 简单平均，而是围绕可见窗口、数据与资源异构、低秩更新语义和知识冲突建立分组、异步、选择性聚合机制。

## 7. 统一评测框架：从算法精度到飞行级闭环

### 7.1 评测维度

| 维度 | 最低报告项 |
| --- | --- |
| 变化与触发 | 漂移类型、检测信号、检测延迟、误报率、漏报率、触发阈值 |
| 更新对象 | RAG、Prompt、Adapter、LoRA、任务头；可训练参数比例、增量大小 |
| 单星资源 | 峰值训练内存、训练时长、能耗、热状态、可见窗口完成率 |
| 新旧能力 | 新任务性能、旧任务性能、平均准确率、forgetting、forward/backward transfer |
| 知识可信度 | Recall@k、grounded answer accuracy、证据一致性、冲突率、索引刷新时延 |
| 星座协同 | 星间/星地字节数、staleness 分布、轨内聚合次数、群组公平性、收敛时延 |
| 发布安全 | A/B 验证结果、灰度比例、版本一致率、回滚时间、失败恢复率 |

### 7.2 必要对比基线

1. 静态冻结模型；
2. 仅 RAG/记忆更新；
3. 仅 Prompt 或任务配置更新；
4. 单星 Adapter / LoRA 更新；
5. 单星更新 + 遗忘控制；
6. 同步聚合与异步聚合；
7. 朴素 LoRA 因子平均与子空间/维度级聚合；
8. 无回滚发布与带灰度验证、回滚的发布策略。

### 7.3 证据边界

当前公开工作中，CLeaRS、RS-RAG、LoRA-Det、FedMosaic、AsyncFLEO 和 GLoRA 分别为问题定义、知识更新、参数高效适配、记忆合并、异步协同和异构 LoRA 聚合提供方法证据；但它们尚未共同构成“在轨 VLM 在线适应—多星协同—安全发布”的飞行级系统证据。[R1][R3][R6][E2][R11][R16]

---

## 8. 章节总结与研究空白

本章应以如下逻辑收束：

> 持续适应的本质不是让所有卫星持续训练，而是使系统能够在变化发生时，先识别变化来源，再以最小可控更新单位恢复能力，并在局部验证通过后进行星座协同和安全发布。面向低轨算力星座的关键研究空白，在于将变化触发、在轨 PEFT、检索记忆更新、异步子空间聚合、版本治理与回滚机制统一到可测量、可验证的飞行级闭环中。
