# 单星部署-综述正文与技术证据

> **文档定位**：本文件是“单星部署：轻量化 VLM 与硬件感知”章节的工作文档，涵盖正文结构、技术证据、引用映射、图表建议与后续调研计划。
>
> **证据口径**：必须区分“地面算法有效”“端侧/飞行代表硬件有效”和“已在轨执行”。NAVI-Orbital 已报告零样本 VLM 在轨自主多模态推理；但除这一类直接飞行演示外，VLM 的视觉 token 控制、KV cache 管理、量化和运行时优化大多仍缺少直接在轨对比验证。写作时必须逐项标注证据层级，不能将地面或端侧结果表述为已获飞行证明。
>
> **正文总论点**：单星 VLM 部署不是将地面模型简单压缩后迁移到卫星，而是在高分辨率遥感输入、有限 DRAM、功率—热控约束、辐射扰动与任务时限共同作用下，对“感知—推理—输出—恢复”全链路进行硬件感知协同重构。

## 0. 章节边界与统一问题表述

### 0.1 本章范围

本章聚焦低轨单星上的视觉—语言模型（VLM）及其相关视觉基础模型部署，讨论：高分辨率输入管理、模型紧凑化、KV/运行时管理、硬件协同、动态资源治理与推理期可靠性。

本章不展开：

- LoRA、Adapter、prompt、联邦聚合与漂移检测等**训练/更新策略**；这些属于“持续适应：在轨微调”章节；
- 星间模型切分、跨星激活传输和星座级调度；这些属于“模型切分与协同推理”章节；
- 地面端遥感模型训练或纯通信网络优化。

### 0.2 单星 VLM 推理闭环

单星部署的目标不是最大化离线吞吐，而是在任务窗口内稳定完成：

```text
输入采集/缓存 → 预处理与 ROI 路由 → 视觉编码 → 跨模态推理/生成
→ 结构化输出或下传 → 异常检测与恢复
```

建议使用以下约束表达式作为本章总纲：

```text
M_peak ≤ M_available,
E_task ≤ E_available,
T_end-to-end ≤ T_deadline,
R_mission ≥ R_required.
```

| 变量 | 含义 | 最低报告项 |
| --- | --- | --- |
| `M_peak` | 推理峰值内存 | 权重、激活、视觉 token、KV、输入缓存、工作区 |
| `E_task` | 单任务能量 | J/image、J/token、J/task、占空比 |
| `T_end-to-end` | 端到端时延 | 预处理、prefill、decode、输出/下传分项 |
| `R_mission` | 任务连续性 | 错误检测、恢复时间、回退成功率、热稳态可用率 |

### 0.3 四类系统瓶颈

| 瓶颈 | 典型来源 | 对 VLM 的直接影响 | 优先技术 |
| --- | --- | --- | --- |
| 视觉工作负载 | 大幅面、多尺度、多时相、多谱段输入 | visual token、prefill 和输入缓存膨胀 | ROI、分块、token 合并/剪枝、流式编码 |
| 峰值内存 | 权重、激活、KV、运行时工作区 | 模型可加载但无法稳定推理 | 分部件量化、结构化压缩、KV 压缩、分页 |
| 持续功率与热控 | 日照/阴影、能量预算、热降频 | 峰值性能不可持续 | 动态分辨率、early exit、模式切换、任务调度 |
| 飞行可靠性 | TID、SEU/SEE、软件异常、存储错误 | 静默错误、任务中断、错误输出 | 校验、检查点、关键算子保护、回退 |

## 1. 正文结构：资源受限单星中的 VLM 推理闭环

### 1.1 从“模型上星”到“任务闭环”

**正文论点。** 星上部署必须以任务闭环为单位评估，而非只比较参数量或 TOPS。视觉编码器、跨模态连接器和语言解码器分别引入视觉 token、权重/激活和 KV cache 压力；模型可被载入 DRAM 并不等价于可在轨持续、可靠地完成任务。

**建议正文顺序。**

1. 说明遥感输入的高分辨率、多尺度、多谱段、多时相特点；
2. 说明 VLM 相较传统 CNN 的新增资源压力；
3. 引入 `M_peak`、`E_task`、`T_end-to-end` 和 `R_mission` 四项约束；
4. 用“任务—资源—硬件—恢复”闭环引出后续技术。

**可直接使用的结论。**

> 单星 VLM 部署首先是端到端资源闭合问题：权重、视觉 token、激活、KV cache、输入缓存、功率/热控与恢复机制必须在同一任务窗口内共同满足约束，而非只比较模型参数量或芯片峰值 TOPS。[R1][R2]

**引用建议。** Sang 等用于星上 RSFM 的资源匹配与软硬件部署链路；Du 等用于“压缩 + 硬件执行 + 域适配”缺一不可的直接空间证据。[R1][R2]

### 1.2 直接在轨 VLM 证据：NAVI-Orbital

NAVI-Orbital 报告其于 **2026 年 4 月 16 日**在低轨航天器上完成零样本本地 VLM 的自主多模态推理演示。系统采用本地 Gemma 3，对新获取的地球观测图像执行场景分类、场景及要素关系描述，并响应操作员的自然语言追问；其运行链路使用图状态机协调检测与对话代理，图像在星上经过 GPU 加速推理处理，且未针对飞行仪器进行微调。[R27]

该工作的重要意义在于：它将“单星可运行 VLM”从端侧或仿真可行性推进到直接飞行证据，说明本地基础模型可通过在轨语义压缩改变“先完整下传、再地面理解”的传统流程。[R27]

但 NAVI-Orbital 不能单独证明某一种 token 压缩、KV 管理、低比特量化或辐射容错方案已被在轨验证。其更适合作为**证据等级 A：已在轨执行**的系统可行性案例；本章其余轻量化与运行时方法仍需分别标注为端侧实测、飞行代表环境验证或地面算法证据。

**可直接写入正文的结论。**

> NAVI-Orbital 已证明零样本 VLM 可以在低轨航天器上完成本地场景理解、文本描述和自然语言交互；但要将一次系统演示发展为可复制的单星部署范式，仍需进一步闭合视觉 token、峰值内存、持续功率、热稳态、故障恢复和跨传感器泛化等指标。[R27]

## 2. 面向星上资源约束的 VLM 执行链重构

> 本章核心节。技术不按“量化、剪枝、蒸馏”平铺，而按模型从输入到输出的执行路径组织。

```text
原始观测
  → 云/模糊/变化/目标前筛
  → 全局缩略图 + ROI 高分辨率裁块
  → token 合并、选择或剪枝
  → 视觉编码器 + 连接器 + 紧凑解码器
  → KV/内存感知运行时
  → 结构化输出、下传、回退
```

### 2.1 输入自适应与视觉 token 控制

#### 正文论点

遥感 VLM 的 token 压缩不应追求均匀削减，而应形成“全局语境保留—局部高价值区域增强—背景与时序冗余压缩”的非均匀资源分配，以同时降低 prefill、输入缓冲和后续多模态前缀开销，并保护小目标、细长目标和空间关系。

#### 技术路线与已核验证据

**图像模块化、裁块与空间组织。** LLaVA-UHD 采用可变尺寸切片、视觉 token 压缩与空间模式组织，支持高分辨率输入。其思路可转化为“全局缩略图 + ROI 高分辨率块 + 空间坐标/尺度编码”的星上输入路径。[R6]

**token merging。** ToMe 合并相似 token，在 ViT-L/H 设置中报告约 2× 吞吐提升、约 0.2–0.3 个百分点的准确率损失。[R5] 星上应将其优先用于背景和多时相冗余区域，而非对 ROI 内 token 采用同等压缩强度。

**预算驱动的 token pruning。** FitPrune 用注意力统计给出固定预算下的训练免费裁剪策略，在 LLaVA-NEXT 上报告 54.9% FLOPs 降低和 0.5 个点准确率损失。[R7] GlimpsePrune 先快速判断图像复杂度再动态裁剪，在其 VQA 设置中报告可裁剪大量视觉 token。[R8]

#### 星上适配判断

- 固定预算裁剪适合任务窗口和功率预算相对确定的场景；
- 动态裁剪适合随电量、温度和输入复杂度变化的运行场景；
- 遥感任务必须额外评估 ROI 漏检、块边界、小目标召回、空间关系理解和多时相一致性；
- 对 SAR、红外、高光谱输入，token 策略还需要纳入谱段/极化/时间维选择，不能直接照搬 RGB VQA 配方。

#### 本节可直接写入正文的结论

> 视觉 token 是高分辨率遥感 VLM 的系统资源变量，而非仅是模型内部表示。单星部署应以任务价值分配 token：用低成本全局视图维持场景语境，用高分辨率 ROI 保护关键对象，并优先压缩低价值背景与时序冗余区域。[R5][R6][R7][R8]

#### 最低报告指标

| 类别 | 最低报告项 |
| --- | --- |
| 输入组织 | 原始分辨率、裁块大小、重叠率、ROI 数量 |
| token | 压缩前后 token 数、ROI token 占比、压缩规则 |
| 任务质量 | 小目标召回、边界误差、空间问答正确率 |
| 运行时 | visual prefill 时延、输入缓冲峰值、J/image |

### 2.2 分部件轻量化：量化、剪枝、蒸馏与紧凑模型

#### 正文论点

VLM 的视觉编码器、跨模态连接器、语言解码器和输出头的资源特征与量化敏感性不同。因此，单星部署不应采用“所有层统一 INT4”式策略，而应使用分部件、混合精度和任务感知的紧凑化配置。

| 模型部件  | 主要约束          | 优先策略                    | 主要风险         |
| ----- | ------------- | ----------------------- | ------------ |
| 视觉编码器 | 高分辨率计算和中间激活   | INT8/混合精度、层级编码、结构化剪枝、蒸馏 | 小目标与空间细节丢失   |
| 连接器   | 跨模态投影与对齐      | 窄连接器、低秩投影、保守量化          | 视觉—语言对齐退化    |
| 语言解码器 | 权重、带宽和 decode | W4/W8、GQA/MQA、小型解码器     | 生成质量与事实一致性下降 |
| 输出头   | 长输出成本         | 结构化头、模板化生成、受限解码         | 开放表达能力降低     |

#### 量化证据

AWQ 按激活统计保护重要通道，并实现 4-bit 端侧 LLM/VLM 推理；SmoothQuant 通过权重—激活等价变换支持 W8A8 推理。[R10][R11]

Q-ViT 指出 ViT 的 attention head 具有不同量化鲁棒性，MSA 与 GELU 是敏感组件；I-ViT 以整数近似 Softmax、GELU、LayerNorm，实现端到端 INT8 ViT 推理，并在其 TVM/GPU 设置中报告实测加速。[R12][R13]

VLMQ 进一步指出，视觉 token 冗余及视觉/文本激活分布差异使通用 LLM PTQ 不能直接等同于 VLM PTQ。[R14]

#### 剪枝、蒸馏与紧凑模型证据

EfficientVLM 将蒸馏与模态自适应剪枝结合，报告更小模型在保持性能下的推理加速；TinyLLaVA 展示了小规模多模态模型可以通过视觉编码器、连接器、语言模型和训练配方的共同选择获得有竞争力的能力。[R9][R15]

Du 等的在轨 GeoFM 工作提供了最重要的空间证据：其压缩版 Prithvi-EO-2.0-300M 从约 1.2 GB 降至约 73 MB，并在 IMAGIN-e 载荷上执行推理；同时，跨传感器域差异会显著影响任务结果，说明压缩不能替代域适配。[R2]

#### 星上适配判断

- 视觉编码器 INT8、连接器保守量化、语言解码器 W4/W8、Softmax/LayerNorm/GELU 保留较高精度，是可作为实验起点的分部件策略；
- 蒸馏和可被硬件识别的结构化剪枝更可能转化为真实资源收益；
- 非结构化稀疏若没有匹配 kernel，不应视为时延或能耗收益；
- 所有量化结果必须在目标硬件上报告回退算子、支持位宽、输入形状、功耗模式和热稳态表现。

#### 本节可直接写入正文的结论

> 单星 VLM 的紧凑化应按视觉编码器、连接器、语言解码器和输出头分别分配精度与容量预算：蒸馏和结构化压缩降低基础容量，混合精度保护量化敏感模块，而目标硬件的算子支持决定压缩能否形成真实物理加速。[R2][R9][R10][R12][R13][R14]

#### 最低报告指标

| 类别 | 最低报告项 |
| --- | --- |
| 配置 | 视觉/连接器/语言/KV 位宽，层数、head 数、参数量 |
| 内存 | 权重、激活、KV、工作区和运行时峰值 |
| 性能 | prefill/decode 分项时延、吞吐、J/image 或 J/token |
| 质量 | 遥感任务精度、生成正确性、幻觉率、置信度校准 |

### 2.3 KV cache、长序列与生成运行时管理

#### 正文论点

生成式 VLM 的峰值内存由视觉前缀、上下文长度、输出长度与 KV 表示共同决定。仅压缩权重无法解决长生成或连续监测任务中的缓存增长；需要从模型结构、缓存表示、内存管理与上下文策略共同治理。

#### 技术路线与已核验证据

**结构级减量。** GQA 以少于 query head 的 KV head 在质量与速度之间折中；其适合作为模型选型条件，而不是任意已训练模型的纯部署后处理。[R16]

**表示级压缩。** KIVI 对 key 和 value 使用不同的分组量化方式，在其 LLM 评测中报告峰值内存降低和吞吐提升。[R17]

**内存管理。** PagedAttention 将 KV 组织为固定大小块，减少动态增长造成的碎片；它的服务器 GPU 结果不可直接等价为星上收益，但按需分块、预分配和缓存复用的运行时原则对有限 DRAM 仍有意义。[R18]

**流式上下文与 prefill 分段。** StreamingLLM 用注意力 sink 与近期窗口处理超长流；MInference 以 head 级稀疏模式加速长上下文 prefill。[R19][R20]

#### 星上适配判断

- 对筛选、分类、短字段告警等任务，优先降低视觉 token 往往比优化 KV 更有效；
- 对问答、解释、灾害报告和连续监测，GQA/MQA、KV 量化、分页与历史摘要是必要路径；
- VLM 需保护系统指令、空间位置描述和关键视觉 anchor token，避免在淘汰策略中误删；
- visual prefill 与 text decode 应分开测量和优化，不能用单一 tokens/s 代替。

#### 本节可直接写入正文的结论

> 单星生成式 VLM 的内存边界不是由参数量单独决定，而是由视觉前缀、上下文、输出长度和 KV 表示共同决定。针对长输出任务，应联合采用结构减量、低比特缓存、块式分配和分层上下文；针对短输出任务，则应优先控制视觉 token。[R16][R17][R18][R19][R20]

#### 最低报告指标

| 类别 | 最低报告项 |
| --- | --- |
| KV 配置 | KV head 数、位宽、最大上下文、最大输出长度 |
| 内存 | KV 峰值、碎片率、缓存回收策略 |
| 时延 | prefill、首 token、decode、端到端时延 |
| 质量 | 输出一致性、事实性、视觉 anchor 保留率 |

## 3. 硬件感知编译、运行时与动态资源治理

### 3.1 从理论压缩率到物理加速比

#### 正文论点

参数量、FLOPs 或稀疏率的下降不会自动转化为星上时延与能耗收益。只有当量化位宽、剪枝结构、动态形状和内存布局能够被目标硬件内核与编译工具链识别时，算法压缩才会成为可测量的物理加速。

```text
模型配置 → 导出/中间表示 → 图优化与校准 → 平台特定编译
→ 内存/功率运行时管理 → 实测时延、能耗、温度和恢复行为
```

Sang 等将星上部署的软件桥梁概括为模型转换、编译与运行时三层；模型可经 ONNX 等中间表示转换，编译阶段处理融合、内存布局和量化，运行时管理资源分配与加速器执行。[R1]

| 平台 | 优先策略 | 主要风险 |
| --- | --- | --- |
| Jetson/GPU | W4/W8、kernel fusion、分块 prefill | 热降频、统一内存峰值 |
| VPU/NPU | INT8、固定图、前端筛选 | 自定义 Transformer 算子受限 |
| FPGA/MPSoC | 定点、流式数据流、结构化模型 | 开发复杂、资源碎片 |
| Rad-hard FPGA | 小模型、强约束数据流、容错优先 | 容量和算力受限 |

Govorkova 等面向辐射硬 FPGA 的 hls4ml 后端展示了硬件感知量化和自动 HLS 转换的闭环。其对象是轻量模型而非 VLM，但可作为“从算法到空间级硬件”的工程先例。[R22]

#### 必须避免的写法

- 不写“FLOPs 降低 x%，所以星上时延降低 x%”；
- 不写“INT4 因此功耗必然降至四分之一”；
- 不用服务器 GPU 吞吐替代 CubeSat、COTS 空间平台或辐射硬平台结果；
- 不用一次冷启动 benchmark 代替热稳态和长时间任务负载。

#### 最低报告指标

| 类别 | 最低报告项 |
| --- | --- |
| 编译 | 支持/回退算子比例、融合策略、动态形状策略 |
| 内存 | 静态分配、峰值、碎片、模型加载时间 |
| 能耗 | 平均/峰值 W、J/image、J/token |
| 热控 | 温度、降频点、稳态吞吐、任务占空比 |

### 3.2 动态资源治理：从 early exit 到任务模式切换

#### 正文论点

early exit、动态分辨率、动态 token 预算和模型回退应统一理解为动态资源治理机制。运行时根据可用电量、温度、DRAM、任务截止期、链路状态、输入复杂度与模型置信度，在不同执行模式之间切换。

| 模式 | 典型任务 | 主要执行组件 | 资源目标 |
| --- | --- | --- | --- |
| M0 | 云/模糊/无效数据筛选 | 规则或轻量视觉模型 | 最低能耗、最大覆盖 |
| M1 | 分类、检测、变化判断 | 视觉编码器 + 小任务头 | 快速感知 |
| M2 | 结构化短报告 | 压缩 VLM + 受限输出 | 告警、属性归纳 |
| M3 | 高价值解释或复杂问答 | 较完整 VLM + KV 管理 | 复杂理解与证据汇总 |

BranchyNet 使用中间分类器进行高置信提前退出；LayerSkip 将早退和自推测解码引入 LLM 推理。[R21a][R21]

SpaceVerse 与 Grace 采用紧凑星上 LVLM 承担低成本任务、地面大模型处理复杂任务，并以置信度和链路条件决定处理路径。这两项工作适合用来说明“星上小模型 + 地面大模型兜底”的任务分级思想，但不应被描述为完整 VLM 已在单星上常态化运行的证据。[R3][R4]

#### 本节可直接写入正文的结论

> 对单星平台而言，early exit 的价值不只是降低单次前向计算，而是将模型深度、输入分辨率、token 预算和输出长度纳入资源状态驱动的任务治理机制。这样，星上系统可在资源正常时执行复杂理解，在资源紧张或任务价值较低时退化为轻量筛选或结构化短输出。[R21][R21a][R3][R4]

## 4. 推理期可靠性与降级运行

### 4.1 故障对象与保护路径

| 故障对象 | 直接风险 | 最低保护机制 |
| --- | --- | --- |
| 权重/LoRA 包 | 模型语义偏移、启动失败 | 哈希、签名、A/B 版本与回滚 |
| 激活/中间矩阵 | 单次推理错误 | ABFT、范围检测、关键层重算 |
| Softmax/GELU/LayerNorm | 非线性误差放大 | 较高精度、范围保护、保守近似 |
| KV cache | 长序列错误累积 | block 校验、关键 token 固化 |
| 编译产物/运行时状态 | 异常退出、错误算子 | watchdog、checkpoint、CPU/规则回退 |

### 4.2 已核验证据

Xue 等针对 ViT 的软错误研究发现，Softmax 和 GELU 等非线性组件比线性 GEMM/全连接部分更脆弱，并提出分块 ABFT 和范围保护方案。[R23]

ReaLM 面向 LLM 推理，先通过故障注入评估容错性，再采用统计 ABFT 和低成本错误检测电路，以较低硬件开销减轻错误导致的推理质量退化。[R24]

这些研究不是空间辐射试验，但为“优先保护高敏感组件、以软件/算法降低冗余”的选择性容错策略提供了技术依据。空间场景还必须叠加 TID/SEE、ECC/EDAC、看门狗复位、存储损坏和不可维修等约束。

### 4.3 单星最小可靠性闭环

```text
加载前：模型/Adapter 哈希与版本验证
运行中：温度、电量、内存、错误计数监控
异常时：checkpoint → 重试 → 轻量模型回退 → CPU/规则回退
恢复后：记录错误、保留任务证据、等待安全版本切换
```

### 4.4 本节可直接写入正文的结论

> 单星 VLM 的可靠性不应只被理解为硬件防护。更可操作的路径是将故障敏感性分析、关键算子保护、模型完整性校验、检查点恢复和任务级降级写入推理运行时，使系统在软错误、热异常或资源不足时优先维持可解释的最低任务能力。[R23][R24]

## 5. 面向单星任务的部署策略选择

| 任务/输入条件 | 推荐模型形态 | 优先优化组合 | 不宜采用的策略 |
| --- | --- | --- | --- |
| 高频数据质量筛选 | 轻量视觉模型或视觉编码器 + 小头 | INT8、固定输入、M0、early exit | 长文本生成、复杂 KV 管理 |
| 大幅面遥感理解 | 分块编码器 + ROI 路由 + 短输出 VLM | 全局缩略图、ROI、token 合并、分段 prefill | 全部原图 patch 平权进入 LLM |
| 高光谱/多时相判别 | 流式编码 + 任务连接器 | 谱段选择、窗口缓存、分层加载 | 一次性拼接全部谱段/时相 |
| 高价值事件解释 | 小型 VLM 或层级调用 VLM | W4/W8、GQA、KV 量化、结构化输出、M2/M3 | 无限制开放生成 |
| 资源/温度异常 | 轻量模型回退或延迟执行 | token 预算收缩、checkpoint、降级模式 | 维持满载长上下文推理 |

**总判断。** 部署策略由任务价值、输入复杂度和可用资源共同决定，而不是由模型参数量单独决定。

## 6. 单星部署评测框架与报告规范

### 6.1 单星部署报告卡

| 维度 | 最低报告项 |
| --- | --- |
| 任务与输入 | 传感器、分辨率、波段/时相数、任务时效目标、输出长度 |
| 模型配置 | 参数量、视觉 token 上限、上下文长度、量化位宽、GQA/KV 配置 |
| 资源效率 | 峰值 DRAM、存储占用、prefill/decode 时延、吞吐、J/image 或 J/task |
| 任务质量 | 精度、小目标召回、边界误差、生成正确性、幻觉率、置信度 |
| 飞行可用性 | 热稳态吞吐、降频曲线、恢复时间、回退成功率、任务可用率 |

### 6.2 推荐对比基线

1. 未压缩或统一 FP16/BF16 模型；
2. 仅权重量化模型；
3. 仅视觉 token 压缩模型；
4. 全链路协同方案：量化 + token 控制 + KV 管理 + 运行时策略；
5. 资源异常或故障条件下的降级方案。

### 6.3 写作边界

- 只报告地面精度或 FLOPs，不能证明方案可在轨部署；
- 只报告模型权重，不报告激活/KV/输入缓存，不能证明内存闭合；
- 只报告峰值吞吐，不报告功率、温度和任务占空比，不能证明持续能力；
- 只报告无故障结果，不报告错误检测与恢复，不能证明飞行可用性。

## 7. 与“持续适应：在轨微调”章节的接口

本章仅讨论推理期的版本加载、校验、回滚和降级。以下内容移交持续适应章节：LoRA/Adapter 的训练策略、QLoRA、联邦 LoRA、漂移检测、异步聚合、星间记忆合并和训练—推理资源竞争。

Pu 和 Xu 的 LoRA-Det 是两章接口案例：它面向星载遥感目标检测，仅更新小部分参数并保持接近全量微调的性能，说明“上传小型参数增量而非全模型”具有明确的星地通信价值。[R25]

Du 等的在轨 GeoFM 研究则说明，跨传感器域差异可显著损害任务表现；因此，“压缩后能运行”不等于“在实际任务域中可靠”，持续域适配仍是必要后续环节。[R2]

## 9. 后续调研计划

### 9.1 优先级

1. **第一优先级**：遥感/高分辨率 token 控制、VLM 量化、KV/长序列、流式 prefill、硬件感知运行时；
2. **第二优先级**：结构化剪枝、蒸馏、稀疏、MoE、early exit；
3. **第三优先级**：飞行级可靠性、故障恢复、动态功率/热控调度；
4. **最后补齐**：遥感小目标、多光谱、多时相和实际在轨案例，特别是持续更新已公开 VLM 飞行演示的硬件、能耗、热控与可靠性指标。

### 9.2 文献与工程资料来源优先级

1. 原始论文的正式发表版本、作者公开版本；
2. 芯片、编译器、运行时与任务方官方技术资料；
3. 真实在轨任务的一手论文、技术报告和任务方公告；
4. 高质量综述仅用于定位脉络；
5. 媒体与博客仅作线索。

### 9.3 核验规则

- 每篇关键论文至少核验：方法对象、实验平台、输入设置、量化/缓存配置、指标和局限；
- 每个“可上星”判断必须标注证据等级：A 已在轨、B 飞行代表环境、C 端侧实测、D 地面算法；
- 性能数值需保留精度格式、输入大小、批大小、上下文长度、功耗模式和硬件；
- 不将地面开发板、后续产品规格或同系列芯片辐照结果直接回填到已发射载荷。
