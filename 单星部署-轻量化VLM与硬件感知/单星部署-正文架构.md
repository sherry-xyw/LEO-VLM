# 单星部署：正文架构

> **章节定位**：本章讨论视觉—语言模型及相关遥感基础模型如何在低轨单星上完成稳定、可验证的本地推理。核心不是将地面模型等比例缩小，而是在高分辨率遥感输入、有限 DRAM、功率—热控约束、辐射扰动、存储寿命与任务时限共同作用下，对输入、模型、运行时、硬件映射和故障恢复进行协同设计。
>
> **核心论点**：单星部署应被建模为“任务需求—资源约束—模型配置—运行时调度—硬件执行—可靠性保障”的闭环。压缩、量化、剪枝、KV cache 管理、视觉 token 控制与 early exit 都是服务于该闭环的技术手段，而非彼此孤立的优化模块。

---

## 1. 问题起点：为什么地面 VLM 不能直接迁移到单星

### 1.1 单星任务闭环受到多重资源约束

对单星而言，部署目标并非单一的模型精度或离线吞吐，而是在给定任务窗口内完成可信感知、语义推理和高价值信息输出。部署可抽象为：

```text
遥感观测采集
  → 预处理、质量评估与 ROI 路由
  → 视觉编码与视觉 token 组织
  → 跨模态推理/文本生成
  → 结构化结果、语义摘要或选择性下传
  → 运行状态监测、故障隔离与恢复
```

这一闭环同时受到以下约束：

```text
M_peak ≤ M_available,
E_task ≤ E_available,
P_avg ≤ P_thermal,
T_end-to-end ≤ T_deadline,
R_mission ≥ R_required.
```

其中，`M_peak` 包括模型权重、视觉特征、KV cache、临时激活和运行时缓冲；`E_task` 和 `P_avg` 分别对应任务能耗和热稳态；`T_end-to-end` 包括输入预处理、推理、后处理与下传决策时延；`R_mission` 则反映对辐射、热波动、文件损坏和软件异常的容错能力。

### 1.2 遥感 VLM 的部署负担主要来自视觉输入、生成式解码和运行时状态

与通用边缘视觉任务相比，星上 VLM 具有三个突出特征：

- **高分辨率与大幅宽观测**使视觉 token 数可能随切片数和输入尺度显著增长；
- **生成式语言解码**会引入随上下文长度和输出长度增长的 KV cache 与自回归时延；
- **任务时效与下传收益耦合**：并非所有观测都需要完整生成，系统应根据任务价值、剩余资源和风险决定推理深度与输出粒度。

LLaVA-UHD 表明，高分辨率图像处理需要同时解决任意宽高比、切片组织与视觉 token 压缩问题；KIVI、PagedAttention 和 MInference 则说明，长序列推理的内存与预填充时延需要从 KV cache 与注意力计算两个层面治理。[R7][R8][R9][R10]

### 1.3 本章与相邻章节的边界

- **单星部署**关注给定模型或模型版本如何在单颗卫星上高效、稳定地执行；
- **持续适应**关注模型、知识库与策略如何在任务期内发生受控更新；
- **模型切分与协同推理**关注模型组件如何在星间或星地分配执行；
- 本章可讨论模型运行时的动态配置，但不将 LoRA、Adapter、联邦聚合或漂移检测作为主体。

## 2. 约束建模与部署决策：先确定任务需要什么，而不是先压缩模型

### 2.1 从任务需求出发建立 Task–Constraint–Model 映射

单星部署的首要问题是：在特定任务、载荷和平台条件下，哪些能力必须保留，哪些计算可以延后、降级或省略。建议以 Task–Constraint–Model（TCM）映射组织部署设计：

| 任务条件 | 关键约束 | 模型/运行时选择 | 优先指标 |
| --- | --- | --- | --- |
| 大幅宽筛查、低事件密度 | 输入带宽与视觉编码成本 | ROI 路由、低分辨率预筛、稀疏切片 | 单位面积计算量、漏检率 |
| 小目标或细粒度辨识 | 空间细节与视觉 token 数 | 多尺度切片、局部高分辨率重编码 | 召回率、token 预算 |
| 开放词汇筛选与语义解释 | 语言解码与外部知识 | 小型 VLM、结构化提示、受控输出长度 | 正确率、生成时延 |
| 多帧或任务日志推理 | 上下文与 KV cache | 流式状态、KV 量化、分页管理 | 峰值内存、首 token 时延 |
| 高优先级告警 | 时效与可靠性 | 早退、任务分级、双路径验证 | 端到端时延、误警/漏警 |
| 平台异常或资源紧张 | 热、能量、存储、链路 | 动态模式切换、降级输出、检查点恢复 | 任务完成率、恢复时间 |

Sang 等对遥感基础模型星载部署的综述强调，数据、模型优化、系统与应用不能割裂考虑；Du 等的在轨 GeoFM 演示进一步表明，模型压缩和飞行环境下的域适应需要与硬件资源和热状态共同验证。[R1][R2]

### 2.2 建议采用任务驱动的多模式执行策略

部署模式不应固定为单一精度、单一分辨率和单一推理深度。可定义：

```text
M0：待机/质量筛查，仅执行观测质量评估与轻量预处理
M1：快速筛选，低分辨率或局部模型输出结构化候选
M2：标准分析，执行压缩 VLM 的完整视觉—语言推理
M3：高价值分析，启用局部高分辨率、较长上下文或复核流程
M4：保护模式，资源/热状态异常时降级推理并优先保障关键任务
```

这种模式切换将模型压缩从静态模型属性扩展为运行时决策：同一卫星可根据事件价值、剩余能量、温度、队列长度和下行窗口选择不同的输入规模、token 预算、解码长度和退出层级。

### 2.3 本节正文应形成的结论

> 单星 VLM 的部署设计应由任务需求和平台约束共同决定。脱离目标识别尺度、任务时限、可用内存和热预算而独立比较参数量或量化位宽，无法形成可飞行的部署方案。

## 3. 感知与模型重构：减少进入语言模型之前的不必要负担

### 3.1 输入治理：从“整幅图像”转向“任务相关视觉证据”

对于高分辨率遥感图像，最有效的计算节省通常发生在视觉编码之前。应以任务价值和空间位置为依据，对输入进行分级处理：

- **质量筛查**：剔除严重云雾、过曝、运动模糊、低信噪比或载荷异常观测；
- **ROI 路由**：通过轻量检测器、变化检测器、地理先验或任务规划器选择候选区域；
- **分层切片**：先使用全局低分辨率上下文定位，再对候选区域执行高分辨率编码；
- **多尺度重采样**：为小目标、细粒度关系与场景语义分别分配不同尺度的视觉 token 预算；
- **结果缓存**：复用连续轨迹、相邻帧或重复地理区域中的视觉特征与任务状态。

LLaVA-UHD 通过图像模块化、视觉 token 压缩和空间组织支持任意宽高比高分辨率输入，说明高分辨率多模态理解不能仅依赖统一缩放；其方法可为遥感切片与 token 预算分配提供通用设计参考，但仍需针对大幅宽和地理坐标关系重新验证。[R7]

### 3.2 紧凑 VLM：优先重构视觉编码器、投影器与语言骨干的组合

单星 VLM 不应简单继承地面大模型的视觉编码器、投影器和语言模型规模。更合理的路线包括：

- 使用小型语言骨干和轻量跨模态投影器；
- 对视觉编码输出进行 Token Merging、重采样或下采样；
- 以任务能力为约束蒸馏视觉—语言对齐，而不是只蒸馏分类 logits；
- 对检测、问答、描述和开放词汇筛选建立不同的模型配置与输出模式；
- 将高成本生成替换为结构化标签、区域坐标、置信度和短摘要，必要时再触发完整对话式推理。

MobileVLM 以小型语言模型、轻量视觉编码与高效投影器面向移动端 VLM 推理，并在 Snapdragon 888 与 Jetson Orin 上报告端侧速度；TinyCLIP 则以跨模态 affinity mimicking 与权重继承支持 CLIP 压缩，说明跨模态蒸馏应保留视觉—语言对齐能力。[R3][R6]

### 3.3 模块级压缩：量化、剪枝和蒸馏应按组件分别设计

VLM 的视觉编码器、语言解码器、跨模态投影器和 KV cache 在数值分布、误差敏感性和运行时作用上不同，因此不宜采用统一压缩方案。

| 组件 | 主要负担 | 优先方法 | 关键风险 |
| --- | --- | --- | --- |
| 视觉编码器 | 高分辨率特征与注意力计算 | ViT 量化、结构化剪枝、token 压缩 | 小目标与边界细节丢失 |
| 跨模态投影器 | 视觉 token 到语言空间映射 | 蒸馏、低秩或轻量映射器 | 视觉—语言对齐退化 |
| 语言骨干 | 模型权重与解码计算 | Weight-only 量化、层级裁剪 | 指令遵循与推理稳定性下降 |
| KV cache | 长上下文、自回归状态 | 低比特量化、分页、滑动窗口 | 长程依赖和缓存误差累积 |
| 输出头/任务头 | 特定任务推理 | 任务头裁剪、结构化输出 | 任务迁移能力受限 |

AWQ 利用激活统计保护少量显著权重通道，提供面向低比特 LLM/VLM 权重量化的硬件友好路线；Q-ViT 则说明视觉 Transformer 的量化需要针对注意力头、尺度和位宽进行专门设计；RemoteTrimmer 进一步给出了面向遥感分类的自适应结构化剪枝实例。[R4][R5][R11]

### 3.4 本节正文应形成的结论

> 单星部署的模型紧凑化应以“视觉证据最小充分性”为目标：先控制输入与视觉 token，再以跨模态蒸馏保持对齐，并针对视觉编码器、语言骨干和 KV cache 分别实施量化、剪枝与状态压缩。统一的全模型压缩策略难以同时保障小目标、空间关系和生成可靠性。

## 4. 运行时资源治理：管理 KV cache、计算路径与任务队列

### 4.1 KV cache 与长上下文状态管理

生成式 VLM 的峰值内存不只由权重决定，KV cache 会随图像 token、历史上下文和输出 token 持续增长。单星环境下，需要将 KV cache 作为一级资源管理对象：

- 按 token 或通道实施低比特 KV cache 量化；
- 将长期任务日志、历史帧和当前视觉证据分离为不同缓存层；
- 对重复前缀、任务模板和共享上下文复用缓存块；
- 对低优先级历史状态采用窗口淘汰、摘要化或外存转移；
- 对并发任务采用显式配额和碎片控制，避免一个长任务挤占关键告警任务。

KIVI 通过对 key 与 value 采用不同的量化粒度实现 2-bit KV cache，在其 LLM 实验中降低峰值内存；PagedAttention 则将 KV cache 以分页方式管理，以减少碎片并支持跨请求共享。这两类技术为星上多任务缓存治理提供方法基础，但其服务器/通用 LLM 假设仍需针对单星内存、文件系统和实时任务重新实现。[R8][R9]

### 4.2 稀疏预填充、流式执行与多帧处理

对于多帧观测、长任务日志和高视觉 token 输入，预填充阶段可能成为主要时延来源。可采用：

- 按注意力头选择稀疏计算模式；
- 在连续轨迹中只对新增或变化区域更新视觉状态；
- 将全局场景描述、局部目标证据和任务历史拆分为分层上下文；
- 采用流式输入与增量输出，使高优先级结果先行可用；
- 对非关键区域以摘要、结构化统计或轻量 token 表示替代原始视觉序列。

MInference 通过识别长上下文注意力中的稀疏模式加速预填充；MMInference 将这种思路扩展到多模态长上下文，并针对模态边界和视频时空局部性设计稀疏注意力。二者可作为单星多帧和长任务上下文的运行时参考，而不是直接视为已适用于卫星硬件的方案。[R10][R12]

### 4.3 动态深度与早退：使计算预算服从任务价值

对于低风险、低复杂度或资源紧张任务，模型不必总是执行完整深度和完整解码。可按任务置信度、剩余能量、时限和热状态选择：

- 视觉编码器早退或低层特征输出；
- 语言模型早退与短答案模式；
- 先由低成本模块给出候选，再由完整模型复核；
- 仅在高价值目标、低置信结果或冲突证据下启用深层推理；
- 将早退阈值与误警/漏警成本联动，而不是仅以 token/s 为目标。

LayerSkip 通过训练期层 dropout 与共享早退头支持 early exit 和 self-speculative decoding，表明在不额外引入独立草稿模型的情况下可减少部分推理开销。其具体加速比例来自通用 LLM 任务，星上 VLM 需要进一步验证视觉输入、任务风险和退出置信度之间的关系。[R13]

### 4.4 本节正文应形成的结论

> 单星运行时优化的核心是将内存、计算和任务队列统一纳入调度。KV cache、输入 token、预填充、解码深度和任务优先级不能独立治理；它们共同决定峰值内存、热稳态与关键任务的端到端时延。

## 5. 硬件映射与可靠性：从“能跑”到“可长期运行”

### 5.1 模型配置必须与硬件数据流共同设计

单星平台可能采用抗辐射 CPU、COTS GPU/NPU、FPGA 或异构组合。不同硬件对低比特矩阵乘、注意力、片上存储、访存模式和故障处理的支持不同，因此模型配置应与硬件映射共同确定：

| 硬件/系统侧问题 | 对模型与运行时的影响 | 可采用的设计原则 |
| --- | --- | --- |
| 片上/外部存储容量 | 限制权重、KV cache、图像 tile 与回放缓冲 | 分层存储、固定配额、流式加载 |
| 矩阵计算与带宽 | 决定量化和 token 压缩收益能否兑现 | 选择硬件可支持位宽与对齐粒度 |
| 异构 CPU/GPU/FPGA | 决定预处理、视觉编码、解码和后处理分工 | 将固定模式算子下沉，将控制逻辑保留在通用处理器 |
| 温度与功率上限 | 限制持续推理和峰值负载 | 模式切换、批量控制、热感知调度 |
| 辐射与文件完整性 | 影响模型权重、缓存和程序状态 | 校验、冗余、检查点、故障隔离 |

空间 FPGA 加速器综述表明，数据流、位宽、片上存储与辐射约束需要共同考虑；SpIRIT/Loris 的在轨 AI 子系统设计则将热变化、辐射、分布漂移和低传输带宽视为星载 AI 的并列任务需求。[R14][R15]

### 5.2 可靠性不应被视为部署后的附加功能

对 VLM 而言，可靠性至少包括：

- **模型与缓存完整性**：权重、量化表、KV cache、词表和任务配置的校验与恢复；
- **热与功率保护**：温度、电压或电量超过阈值时自动降级至 M0/M1 模式；
- **任务隔离**：长上下文或低优先级任务不能耗尽关键告警任务的资源；
- **结果可信度**：对低置信、冲突或超出分布结果输出“不确定”或转入复核，而非强制生成；
- **检查点与回退**：推理过程中断后应恢复到稳定任务状态，而不是重新加载完整模型和历史上下文。

Du 等在 ISS 上的 GeoFM 演示表明，压缩模型在飞行环境中的稳定执行还需要处理热状态等平台问题；NAVI-Orbital 则显示本地 VLM 可以在低轨航天器上执行场景分类、文本描述和自然语言交互，但不应将其零样本推理演示外推为对量化、KV cache 或早退策略的在轨验证。[R2][R16]

### 5.3 本节正文应形成的结论

> 单星部署的硬件感知并非选择某一块“更快的芯片”，而是使模型位宽、token 预算、缓存策略、计算图、热管理和容错机制共同适应目标平台。缺少可靠性设计的低时延结果，不构成可长期运行的在轨能力。

## 6. 部署验证与报告规范：从地面算法到飞行级证据

### 6.1 建议采用分阶段验证流程

单星 VLM 的部署验证应遵循由模型能力到任务闭环逐级收紧的流程：

```text
任务与平台约束定义
  → 地面模型压缩与任务适配
  → 端侧/飞行代表硬件 profiling
  → 热、功率、存储与异常注入测试
  → 长时运行与任务队列压力测试
  → Flatsat/飞行代表环境验证
  → 在轨推理与地面结果一致性检查
```

Du 等的 GeoFM 工作覆盖压缩、任务适配、代表环境验证和 ISS 在轨执行，是“模型—硬件—热状态—在轨一致性”闭环的核心案例；NAVI-Orbital 则补充了零样本 VLM 在轨自主多模态推理的直接证据。[R2][R16]

### 6.2 最低报告指标

| 维度 | 最低报告项 |
| --- | --- |
| 模型配置 | 参数量、位宽、视觉 token 数、上下文长度、输出长度、可训练/不可训练模块 |
| 任务性能 | 主任务精度、漏检率、虚警率、结构化输出正确率、语义解释一致性 |
| 输入治理 | 原始分辨率、切片策略、ROI 覆盖率、每任务 token 预算、信息损失 |
| 运行时 | 首 token 时延、每 tile/每帧时延、吞吐、预填充时延、KV cache 峰值 |
| 资源 | 峰值内存、平均/峰值功耗、热状态、存储占用、任务窗口完成率 |
| 可靠性 | 连续运行时间、异常注入结果、恢复时间、缓存/权重完整性、降级策略 |
| 任务闭环 | 高价值信息压缩率、下传节省、告警时效、任务成功率 |

### 6.3 必要对比基线

1. 原始模型与紧凑模型；
2. 仅权重量化与组件级联合量化；
3. 全图编码与 ROI/分层切片；
4. 固定 token 预算与任务感知 token 预算；
5. 全精度 KV cache 与量化/分页/窗口管理；
6. 固定深度推理与 early exit/动态模式；
7. 无热保护/恢复与带热感知调度、检查点恢复的运行时；
8. 地面 GPU 结果、飞行代表硬件结果与在轨执行结果。

### 6.4 证据边界

当前公开研究提供了互补而不等价的证据：MobileVLM、AWQ、Q-ViT、TinyCLIP、LLaVA-UHD、KIVI、PagedAttention、MInference 与 LayerSkip 主要证明边缘或地面环境下的模型与运行时机制；Du 等和 NAVI-Orbital 提供在轨基础模型推理证据；SpIRIT/Loris 与空间 FPGA 研究提供硬件与可靠性约束证据。将这些要素整合为“高分辨率 VLM、长上下文运行时、硬件感知量化与可靠性保障”的完整飞行级系统，仍需要统一实验协议和在轨验证。[R2]–[R16]

## 7. 章节总结与研究空白

本章可归纳为：

> 单星部署的核心不是把 VLM 静态压缩到某个参数量，而是在任务时限、输入规模、DRAM、功率—热控与可靠性约束下，使视觉证据选择、模型紧凑化、KV cache 管理、动态深度、硬件映射和故障恢复形成可协同优化的执行闭环。

仍待解决的关键问题包括：

1. 如何为遥感大幅宽输入建立与任务收益直接相关的视觉 token 预算和 ROI 路由策略；
2. 如何在视觉编码器、投影器、语言骨干和 KV cache 上实施统一但组件敏感的低比特化与结构化压缩；
3. 如何将早退、稀疏注意力、流式缓存与热感知调度纳入同一运行时控制器；
4. 如何将辐射、热波动、存储损坏、任务抢占和模型不确定性纳入 VLM 推理可靠性评测；
5. 如何在真实星载硬件上报告端到端任务闭环指标，而不仅是单模型精度或桌面 GPU 吞吐。
