## 0. 综述定位与差异化（动笔前必须先想清楚）

### 0.1 体裁定位：这是一篇"议程定义型"综述，不是"成果汇编型"综述

写之前必须先认清一个事实：**到 2026 年，真正在轨运行过的卫星视觉智能，绝大多数还是 CNN/小模型（Φ-sat 系列），VLM/多模态大模型在轨基本停留在纸面设计和地面仿真阶段。** 这个领域目前是**高度前瞻（anticipatory）的**——东西还没大规模落地。

这决定了本综述的体裁。综述大致分两类：

- **成果汇编型**：领域已经成熟、成果多，综述的价值是把已有结果整理归类、做横向对比。本领域**还不到这个阶段**，硬走这条路会显得"内容单薄、强行凑数"。
- **议程定义型（本综述选这条）**：领域正在起步、即将爆发，综述的价值是**在领域碎片化之前，把问题空间、决策准则和评价标准先立起来**——类比联邦学习、边缘 AI 早期那批奠基性综述。

把体裁讲清楚有两个直接好处：

1. **把"东西少"这个弱点反转成立项理由**：正因为还没定型，现在统一框架才有意义、才能被后来者反复引用。
2. **提前化解审稿人最可能的攻击**——"这综述是不是写早了？是不是 VLM 蹭热点？"。我们用一张诚实的成熟度地图（见 Ch.2.5）正面回应：领域确实早期，但下行带宽剪刀差（见 Ch.1）使它**必然到来**，因此现在定框架正当其时。

### 0.3 三条不可让步的差异化主张

1. **VLM 优先，不是 LLM 优先。** 低轨载荷绝大多数是图像（光学/SAR/红外/高光谱），文本只是接口语言。综述必须把三条 VLM 特有的主线贯穿全文：(a) **视觉 token 爆炸**——一张大幅遥感图会被切成几千个视觉 token，挤爆上下文窗口；(b) **跨模态对齐在压缩下退化**——连接视觉与语言的投影模块在量化/低秩后特别脆弱；(c) **遥感图像分辨率与上下文长度的矛盾**。这些问题纯文本 LLM 综述里没有，是我们独占的切入点。
2. **以"部署-可靠性-持续性"三角组织，不是算法目录。** 不要按"压缩/联邦/MoE/RAG"逐项罗列，而是对每类空间任务问一句："这三角里，哪一角才是它真正的瓶颈？"
3. **明确硬件约束分三个等级，对应能力天差地别。** (a) 实验室商用边缘硬件（Jetson/FPGA 评估板）；(b) 商业空间 COTS 平台（HPE Spaceborne、NVIDIA Space-1 类）；(c) 抗辐射加固（rad-hard）硬件。三档能跑的模型规模和该用的可靠性策略完全不同，混为一谈是常见错误。

## 2. 章节架构（正文 11 章：引言 + 9 个主体章 + 结论）

### Ch. 1 — Introduction（约 3 页）

**1.1 为什么这个领域必然到来（前景论证，不能只是断言）**

开篇要用一条**量化的硬逻辑**立住"在轨处理是刚需"，而不是泛泛说"产业拐点"：

> **下行带宽剪刀差。** 现代低轨遥感卫星的成像数据生成速率，远快于它能下传的速率。单颗高分卫星每天产出可达 TB 量级，而一次过地面站窗口通常只有约 10 分钟、X 波段下行能力在几百 Mbps 到几 Gbps 量级。两条曲线（产数速率 vs 下行容量）随着传感器分辨率提升越拉越开——这就是"剪刀差"。其后果是：**大量原始数据根本传不下来，只能在轨先做取舍。**

这条逻辑自然推出两步结论：(1) 取舍必须在轨做 → 需要在轨智能；(2) 取舍越来越依赖**语义判断**（"这张图里有没有值得传的东西""发生了什么事件"），而不只是阈值/检测 → **这正是 VLM 的用武之地**。前景论证就立在这条链上。

**1.2 范围声明（关键，必须写死）**

本综述聚焦 **VLM/多模态基础模型在 LEO 单星与算力星座上的在轨智能**。明确**不**涵盖：GEO/深空、纯通信网络优化、纯文本 LLM 应用、地面端遥感大模型训练。

### Ch. 3 — Task-Constraint-Model (TCM) 三元映射 ⭐ 核心理论贡献

> 这是与已有综述拉开档次的章节，目标是写成"未来 3 年这个领域的人都要引用"的框架。本章先回答"该不该上星"（3.1），再回答"若上星、被什么约束住、用什么模型"（3.2–3.5）。

**3.1 决策边界：一个任务该不该上星推理？（新增，本章的"前置门"）**

这是 TCM 之前必须先回答的问题，也是本综述最可能被反复引用的部分。两条处理路径二选一：

- **路径 A（下传地面）**：原始数据下传，在地面服务器处理。代价 = 占用稀缺下行带宽 + 等待下一次过站的时延 + （地面算力很便宜，可忽略）。瓶颈是下行窗口和数据量。
- **路径 B（在轨推理）**：在卫星上跑 VLM，只下传结果（标签、告警、挑出的关键图块）。代价 = 在轨算力的能耗与热 + 受限模型带来的精度损失；收益 = 时延低（不必等过站）、下行量大幅压缩。

引入几个量来判断哪条路径划算：

- $V_{in}$：输入数据量（每次成像 / 每次过站）；
- $V_{out}$：任务有用输出的数据量（告警只有几个字节；要重建整幅图就≈$V_{in}$）；
- $\rho = V_{out}/V_{in}$：**语义压缩比**，越小说明"有用信息相对原始数据越少"；
- $B$：下行容量（每次过站可传比特数）；
- $W_{revisit}$：过站/重访间隔，决定地面路径的时延下限；
- $L^*$：任务可接受的时延上限（deadline）；
- $E_{compute}$：在轨完成该任务的能耗；$P_{budget}$：卫星能量预算；
- $Q_{loss}$：受限的在轨模型相对地面大模型的精度损失。

**在轨推理胜出的判据（决策启发式，非精确定律）：**

> 当 **(1) 下传传不完**（$V_{in} > B\cdot W$，剪刀差成立）**或 (2) 时效等不起**（$W_{revisit} > L^*$，等下次过站就超期，典型如灾害告警）成立，**且 (3) 输出可大幅压缩**（$\rho \ll 1$）**且 (4) 能耗扛得住、精度损失可接受**（$E_{compute}$ 在 $P_{budget}$ 余量内、$Q_{loss}$ 在任务容忍范围）——此时上星推理划算。
>
> 反之（输出≈输入、无时效压力、能耗吃紧、精度要求极高），应老老实实下传地面。

**对应图 F2-a：决策边界图。** 横轴语义压缩比 $\rho$（或时效要求 $L^*$），纵轴数据/带宽比 $V_{in}/(B\cdot W)$，画出 break-even 曲线，把右上区标"该上星"、左下区标"该下传"，再把 8 类任务 T1–T8 落到图上。例如 T3（灾害告警）落在强烈"该上星"区，T1（高质量报告生成、需完整图存档）更偏"下传"区。

**3.1 与 3.2–3.5 的关系（必须讲清，否则读者会混）：** 3.1 是**门**——先判断该不该上星；通过这道门的任务，再进入 TCM 的设计空间（3.2–3.5）去问"被什么约束住、用哪些模型组件"。

**3.2 任务维度（T）：8 类典型空间 VLM 任务**

- T1 遥感图文描述与报告生成
- T2 目标检测 + 自然语言解释
- T3 灾害/事件识别与告警
- T4 变化检测与时空推理
- T5 多源遥感融合问答
- T6 星务/遥测异常诊断
- T7 任务规划与操作建议
- T8 智能下传决策（哪些数据传、哪些丢）

**3.3 约束维度（C）：统一约束-QoS 框架** ⭐（TCM 的"硬核"：把约束轴从标签升级为可形式化、可测量的模型，并与 Ch.9 指标一一对应）

**统一目标（概念性，不是要精确求解）。** 把一次空间 VLM 任务的部署/推理写成：在约束集 $\mathcal{C}=\{C1,\dots,C7\}$ 下，最大化

$$\max\ \sum_k U_k(\hat y_k)\ -\ \lambda_E\, E_k\ -\ \lambda_D\,\mathrm{DeadlineViol}_k$$

**逐项白话解释：** $U_k$ 是任务 $k$ 的**有用产出价值**（准确率 / mAP / 告警价值，刻意用任务级指标，不退化成 token 级代理）；$E_k$ 是能耗、$\mathrm{DeadlineViol}_k$ 是超期程度，二者带负号作为惩罚；$\lambda_E,\lambda_D$ 是权重。**这个式子不是要被解出来，而是给全综述一个统一坐标系**——让 Ch.5–8 的每个方法都能说清"它在放松哪一类约束、或在优化目标里的哪一项"。

**七类约束：**

- **C1 算力/显存**：对 prefill 计算量、decode 步数、KV cache 增长、视觉 token 数**分别**建模；约束 = 节点能承载的并发量。
- **C2 能耗**：J/token、J/image、J/task；含日照-阴影相位的电量预算（能耗-碳建模可对标 LLMSpace, arXiv 2605.05615 —— ⚠️ 该文献待核验，见 §1.1，未确认前不写死此引用）。
- **C3 热控**：真空中只能靠辐射散热；持续/突发负载下结温须 ≤ 阈值，超限就降频、反馈回 C1（VLM 处理大图 prefill 会产生瞬时功耗尖峰，见 Ch.4.5）。
- **C4 通信**：星地过站窗口 + 星间时变拓扑容量；转发只能走当前可见链路。
- **C5 轨道动力学**：这是**确定性**的时变（TLE 已完全确定未来 N 小时的可见性）——**应当作优化变量来用，而不是当干扰**，这是 LEO 区别于地面边缘的根本特征。
- **C6 辐射-故障**：TID 长期累积退化 + SEE/SEU 瞬态翻转；作为**鲁棒性约束**（故障注入下的优雅降级，详见 Ch.8）。
- **C7 安全与可控**：关键任务要设最低精度下限、可解释下限、决策边界与回滚机制。

> 这套约束贯通三处，保证全综述只有**一个自洽框架**：3.5 的 T×C 热力图用它定位瓶颈；Ch.5/6/7/8 各方法按"缓解了哪一(几)类约束"归类；Ch.9 指标体系是它的**可测量化**（每条指标对应一类约束）。

**3.4 模型维度（M）：4 个层级**

- M1 视觉编码器（地理空间基础模型与压缩）
- M2 跨模态桥接（投影器、Q-Former、视觉 token 压缩）
- M3 语言模型主干（小型 LLM 的量化与稀疏）
- M4 任务头与外部记忆（Adapter、RAG、规则校验）

**3.5 映射矩阵：哪一格紧张？哪个组件被压住？**

- 用 8×7 的 T×C 热力图揭示瓶颈分布——这是本综述的**招牌图（F2-b）**，必须打磨数轮。
- 举例：T3（灾害告警）× C4（通信窗口）紧张到极致，瓶颈推向 M4 的下传决策；T7（任务规划）× C7（安全）紧张，瓶颈推向 M3 + 规则校验。

### Ch. 4 — VLM 特异性瓶颈分析 ⭐ 第二大原创点

> 本章独立成章，是为了让审稿人立刻看到"为什么必须把 VLM 当 VLM 看，不能直接套用 LLM 综述的结论"。本章只讲**问题**；具体**方法**留给 Ch.5/Ch.8（写作时反复 diff，避免重复）。

- **4.1 视觉 token 爆炸**：一幅大尺寸遥感图会被切成数千个视觉 token，直接挤爆上下文窗口——这是和上下文长度的硬冲突。
- **4.2 跨模态对齐在带宽受限下退化**：连接视觉与语言的投影模块（projector）在量化/低秩压缩后特别脆弱，先于主干崩坏。
- **4.3 KV cache 体积**：长 prompt 场景下 KV cache 线性膨胀，撞上星上 SRAM/HBM 的容量天花板。
- **4.4 prefill 与 decode 的不对称**：prefill 计算密集、decode 访存密集，二者在能耗/时延上不对称；星载短任务 vs 长任务的设计权衡因此不同。
- **4.5 VLM 推理的热瞬态特性**：突发的大图 prefill 会造成芯片瞬时功耗尖峰，进而触发 C3 热约束的降频反馈。

**4.6 VLM 必要性边界：什么时候非 VLM 不可，什么时候小模型就够（新增）**

这是必须正面回答的边界问题——不回答，审稿人会怀疑整篇是 VLM-hype。

- **小模型（检测/分类 CNN）就够的场景**：闭集、固定类别、纯检测/计数（"有没有船""云量多少"）。这类任务上 VLM 是过度工程，徒增能耗。
- **VLM 真正不可替代的能力**：
  - (a) **开放词表 / 零样本**：地面没预定义过的新目标、突发事件的描述；
  - (b) **指令可重配**：上行一句自然语言就能改任务，不必重训、重传整个模型；
  - (c) **跨模态语义推理与报告生成**：如"描述这片区域刚发生了什么"；
  - (d) **多源遥感融合问答**。
- **判据**：当"任务集合在轨期间会变 / 类别开放 / 需要自然语言交互"时，VLM 的额外成本才划算；否则应退回小模型。
- **诚实提醒**：综述要明确指出，不少被包装成"星载 VLM"的工作其实用检测器就能完成——把这条边界讲清楚，反而显出我们判断力，增强可信度。
