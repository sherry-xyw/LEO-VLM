## 0. 已核验结论与正文写作口径

### 0.1 可作为本章核心结论的判断

1. **单星部署首先是端到端资源闭合问题。** 权重、视觉 token、激活、KV cache、输入缓存、功率/热控和恢复机制必须在同一任务窗口中共同满足约束，而非只比较参数量或峰值 TOPS。[R1][R2]
2. **视觉 token 控制应与量化并列为优先技术。** 遥感输入的大幅面、多尺度和局部稀疏信息特征，使均匀缩放或统一裁剪容易丢失小目标、细长目标和空间关系；更合理的路径是“全局上下文保留—ROI 增强—背景压缩”。[R5][R6][R7][R8]
3. **低比特部署不应采用所有模块统一位宽。** 视觉编码器、跨模态连接器、语言解码器和输出头的量化敏感性不同；VLM 专用 PTQ 研究已指出视觉 token 冗余与视觉/文本激活分布差异会显著恶化通用 LLM 量化。[R10][R11][R12][R13][R14]
4. **KV cache 的优先级取决于输出形态。** 对筛选、分类、短字段告警等短输出任务，先控制视觉 token 往往更有效；对问答、解释、长报告和连续监测，GQA/MQA、KV 量化、分页与流式上下文成为峰值内存与 decode 时延的必要优化。[R16][R17][R18][R19][R20]
5. **理论压缩率不是星上收益。** 只有当位宽、剪枝结构、算子图、内存布局和动态形状被目标硬件及编译工具链支持时，参数/FLOPs 的下降才可能转化为实际时延、能耗和热稳态吞吐的下降。[R1][R13][R22]
6. **可靠降级是单星部署的必要组成。** 辐射和软错误会改变参数、激活、KV 或计算结果；ViT 的非线性模块尤其需要专门保护。应把校验、检查点、A/B 回滚和轻量模型回退写入部署流程。[R23][R24]

### 0.2 推荐的证据标签

| 标签 | 含义 | 正文写法 |
| --- | --- | --- |
| A | 已在轨执行 | “已在轨演示” |
| B | 飞行代表硬件/任务工程验证 | “在飞行代表环境中验证” |
| C | 端侧或嵌入式硬件实测 | “在端侧平台上报告” |
| D | 地面算法实验 | “在地面实验中表明” |

### 0.3 已确认的直接空间证据

| 证据 | 结论 | 标签 |
| --- | --- | --- |
| Prithvi GeoFM 压缩与 IMAGIN-e 在轨执行 | 压缩和域适配是基础模型上星的前提 | A/B |
| LoRA-Det | 小参数更新可显著降低星地更新负担 | D/任务导向 |
| SpaceVerse 与 Grace | 小型星上模型 + 地面大模型是现实协同边界 | D/系统仿真 |

Du 等将 Prithvi-EO-2.0-300M 的一个变体从约 1.2 GB 压缩到约 73 MB，并在 IMAGIN-e 载荷上执行 GeoFM 推理；该工作还显示跨传感器域差异会显著影响任务表现，因此仅做压缩并不能解决实际在轨适配问题。[R2] 这项证据支撑“压缩 + 域适配 + 硬件执行”三者必须共同出现，但其对象是 GeoFM/ViT，而不是生成式 VLM。

## 1. 输入自适应与视觉 token 控制

### 1.1 技术定位

视觉 token 控制对应单星 VLM 的**视觉工作负载**：输入分辨率越高，视觉编码器计算、输入缓冲和投影到语言模型的前缀长度通常越大。对遥感任务，关键不是盲目降低分辨率，而是以任务价值分配 token 预算。

推荐的单星输入路径为：

```text
原始图像/多时相数据
  → 云/模糊/变化/目标前筛
  → 全局缩略图
  → ROI 高分辨率裁块
  → token 合并或选择
  → 视觉编码与跨模态投影
```

### 1.2 可直接使用的技术路线

#### A. 图像模块化、裁块与空间组织

LLaVA-UHD 使用图像模块化，将原始分辨率图像划分为可变尺寸切片，再压缩视觉 token 并以空间模式组织切片 token。其在 LLaVA-1.5 336×336 基础上支持 672×1088 输入，报告的推理计算量为基线的 94%，TextVQA 准确率提高 6.4 个点。[R6]

**星上适配判断**：

- 可直接借鉴“全局缩略图 + 局部裁块 + 空间编码”的结构；
- 不应直接复制其通用视觉问答设定，需要评估裁块重叠、坐标/尺度编码和跨块关系；
- 对舰船、车辆、道路、河流等小目标或细长目标，ROI 漏检与块边界是主要风险。

#### B. token merging

ToMe 通过合并相似 token 提升 ViT 吞吐，在 ViT-L 512 和 ViT-H 518 上报告约 2× 吞吐提升，同时准确率下降约 0.2–0.3 个点。[R5]

**星上适配判断**：

- 适合作为背景区域压缩和多时相冗余压缩的基础模块；
- 不宜对 ROI 内 token 使用与背景相同的合并强度；
- 应报告合并前后 token 数、ROI 覆盖率、小目标召回和跨块一致性。

#### C. 预算驱动的视觉 token pruning

FitPrune 通过小批量推理数据的注意力统计生成给定预算下的训练免费 pruning 配方；在 LLaVA-NEXT 上报告 FLOPs 降低 54.9%、准确率下降 0.5 个点。[R7] GlimpsePrune 采用先“glimpse”再裁剪的动态策略，报告可裁剪 92.6% 视觉 token 而维持自由形式 VQA 的平均基线性能。[R8]

**星上适配判断**：

- FitPrune 的“预算固定”逻辑适合任务窗口固定、资源可预估的卫星模式；
- GlimpsePrune 的“复杂场景少裁、简单场景多裁”适合能量/温度状态随轨道变化的动态运行；
- 两者均需在遥感小目标、边界敏感分割、多光谱和多时相数据上重新校准。

### 1.3 本节可直接写入正文的结论

> 对高分辨率遥感 VLM，视觉 token 不是单纯的模型内部变量，而是决定 prefill 计算、输入缓存和后续 KV 状态规模的系统资源变量。因此，单星部署应采用任务感知的非均匀 token 管理：以低成本全局视图保持场景语境，以高分辨率 ROI 保护关键目标，并优先压缩低价值背景和时序冗余区域。[R5][R6][R7][R8]

### 1.4 最低报告指标

| 类别 | 最低报告项 | 目的 |
| --- | --- | --- |
| 输入 | 原始分辨率、裁块大小、重叠率 | 复现空间输入路径 |
| token | 压缩前后 token 数、ROI token 占比 | 定量资源收益 |
| 任务质量 | 小目标召回、边界误差、空间问答正确率 | 防止只看平均准确率 |
| 运行时 | visual prefill 时延、输入缓冲峰值 | 反映真实星上负担 |

## 2. 分部件轻量化：量化、剪枝、蒸馏与紧凑模型

### 2.1 量化：从通用 LLM 到 VLM 特异配置

#### A. 权重量化与激活量化

AWQ 以激活统计识别重要通道，对少量显著权重进行保护；其论文同时实现了面向 4-bit 端侧 LLM/VLM 的 TinyChat，报告相较 HuggingFace FP16 在桌面和移动 GPU 上超过 3× 加速。[R10] SmoothQuant 通过等价变换将激活离群值的量化困难转移到权重端，在 LLM 上实现 W8A8，报告最高 1.56× 加速和 2× 内存降低。[R11]

**星上适配判断**：

- AWQ 更适合作为语言解码器的 W4 起点；
- SmoothQuant 更适合作为需要激活整数化、且目标硬件可有效执行 INT8 GEMM 的路径；
- 两者的原始结果主要来自地面 GPU，星上必须报告目标硬件上的 kernel 支持、量化格式、回退算子与真实能耗。

#### B. 视觉编码器的量化敏感性

Q-ViT 表明 ViT 的不同 attention head 具有不同量化鲁棒性，并指出 MSA 与 GELU 是关键量化敏感部分。[R12] I-ViT 以整数近似 Softmax、GELU 和 LayerNorm，实现端到端 INT8 ViT 推理；在 TVM/GPU 实测中报告相对浮点模型 3.72–4.11× 加速。[R13]

**星上适配判断**：

- “视觉编码器 INT8、连接器保守量化、语言解码器 W4/W8、Softmax/LayerNorm/GELU 保留较高精度”可作为正文推荐的初始配置；
- 对遥感任务，应使用云遮挡、低照度、季节变化、密集小目标、传感器差异等样本进行校准；
- 不应将视觉编码器与语言解码器统一设置为同一极低位宽。

#### C. VLM 专用 PTQ

VLMQ 指出 VLM 中存在视觉 token 过度表示和视觉/文本隐特征分布差异，并用 token 显著性驱动的 PTQ 进行优化；其在 0.5B–32B VLM 与 8 个 benchmark 上的 2-bit 设置中报告 MME-RealWorld 指标提升 16.45%。[R14]

**正文使用边界**：VLMQ 直接说明 VLM 量化不能简单照搬 LLM PTQ；但它仍是预印本，且没有星上硬件结果，宜用于“VLM 特异量化机制”的论证，不宜作为“已适配卫星”的证据。

### 2.2 结构化剪枝与蒸馏

EfficientVLM 先蒸馏再进行模态自适应剪枝，得到 93M 参数的模型（教师规模的 44.3%），保留教师 98.4% 性能并报告 2.2× 推理加速。[R9] TinyLLaVA 的系统研究表明，小规模多模态模型通过更合适的视觉编码器、连接器、语言模型、数据和训练配方，可以达到与较大模型相近的性能；其 3.1B 版本在论文对比中优于若干 7B 基线。[R15]

**星上适配判断**：

- 蒸馏是最具飞行相关性的模型压缩路线之一：GeoFM 的实际在轨工作也采用压缩与蒸馏思想；[R2]
- 剪枝应优先选择通道、层、head、token 等可被编译器和硬件真实识别的结构；
- 非结构化稀疏在没有匹配 kernel 的情况下，不应直接声称节省时延或能耗。

### 2.3 本节可直接写入正文的结论

> 单星 VLM 的紧凑化不应以统一低比特或单一参数压缩率为目标，而应按视觉编码器、跨模态连接器、语言解码器和输出头分别确定资源预算。蒸馏和结构化压缩负责降低基础容量，混合精度负责保护量化敏感模块，而目标硬件的算子支持决定这些压缩能否形成真实物理加速。[R2][R9][R10][R12][R13][R14]

### 2.4 最低报告指标

| 类别 | 最低报告项 | 目的 |
| --- | --- | --- |
| 配置 | 视觉/连接器/语言/KV 的位宽 | 显示分部件策略 |
| 内存 | 权重、激活、KV、工作区峰值 | 避免仅报告权重 |
| 性能 | 端到端时延、prefill/decode 分项 | 区分计算瓶颈 |
| 质量 | 遥感任务精度、生成正确性、置信度 | 评估压缩损失 |

## 3. KV cache 与长序列运行时管理

### 3.1 结构级减量：GQA/MQA

GQA 使用少于 query head 数的 KV head，是 MHA 与 MQA 之间的折中；论文显示其质量接近多头注意力，速度接近 MQA。[R16]

**星上适配判断**：GQA/MQA 需要在基座模型阶段就具备对应结构，不能作为任意已训练模型的纯部署后处理。对于生成式 VLM，它从源头减少 KV cache 容量和 decode 阶段带宽压力，适合作为模型选型条件。

### 3.2 表示级压缩：KV 量化与重要性保留

KIVI 对 key 采用按通道分组量化、对 value 采用按 token 分组量化，在论文评测中以 2-bit KV 量化实现 2.6× 更低峰值内存（含权重），并提升实际 LLM 推理吞吐。[R17]

**星上适配判断**：

- 适合长输出、长上下文和多轮任务；
- 对短字段告警，其收益可能小于视觉 token 控制；
- VLM 中应显式保护系统指令、空间位置描述和关键视觉 anchor token，避免把对定位/解释关键的上下文误淘汰。

### 3.3 内存管理：分页与块式分配

PagedAttention 将 KV cache 组织为固定大小块，以减少动态增长中的碎片与冗余；vLLM 论文在服务器 GPU 上报告在相同延迟下 2–4× 吞吐提升。[R18]

**星上适配判断**：该结果不能直接等价为星上收益，但“按需分块、预分配、避免频繁重分配、共享可复用前缀”的运行时思想对有限 DRAM 仍然成立。单星更应将分页块大小、最大输出长度和上下文回收策略设为可配置任务参数。

### 3.4 流式上下文与 prefill 分段

StreamingLLM 通过保留注意力 sink 和近期窗口，让有限窗口模型处理超长流；其报告相较滑窗重计算基线最高 22.2× 加速。[R19] MInference 通过为不同 attention head 选择稀疏模式加速长上下文 prefill，在 A100 上报告最高 10× prefill 加速。[R20]

**星上适配判断**：

- StreamingLLM 的“初始 token + 最近窗口 + 历史摘要”框架适合连续监测和多时相任务；
- MInference 的绝对性能结果来自 A100，不能直接迁移到 Jetson/FPGA/VPU；但它提示应将 visual prefill 和 text decode 分开建模与优化；
- 对图像/视频式输入，块间摘要和关键时相记忆应成为流式状态设计的一部分。

### 3.5 本节可直接写入正文的结论

> 生成式 VLM 的峰值内存不是由参数量单独决定，而是由视觉前缀、上下文长度、输出长度和 KV 表示共同决定。单星部署应以“模型结构减量—低比特缓存—块式内存管理—分层上下文”组成联合路径，并将短输出任务与长生成任务分开设计。[R16][R17][R18][R19][R20]

## 4. 硬件感知编译与可测量的物理加速

### 4.1 正确的部署链

```text
模型配置
  → 导出/中间表示
  → 图优化与量化校准
  → 平台特定编译
  → 运行时内存与功率管理
  → 实测时延、能耗、温度和恢复行为
```

Sang 等将星上部署的软件桥梁归纳为转换、编译和运行时三层：模型可经 ONNX 等中间表示转换，编译阶段负责算子融合、内存布局与量化，运行时负责内存分配、并行和加速器资源管理。[R1]

### 4.2 平台匹配建议

| 平台类型 | 优先策略 | 主要风险 |
| --- | --- | --- |
| Jetson/GPU | W4/W8、kernel fusion、分块 prefill | 热降频、显存峰值 |
| VPU/NPU | INT8、固定图、前端筛选 | 自定义 Transformer 算子受限 |
| FPGA/MPSoC | 定点、流式、结构化模型 | 开发复杂、资源碎片 |
| Rad-hard FPGA | 小模型、强约束数据流 | 容量与算力受限 |

Sang 等指出，TensorRT 的内核自动调优、层融合和 INT8 校准适用于 GPU 推理；Vitis AI 适合将量化模型映射至 FPGA DPU，但面对新型 Transformer 自定义算子可能需要额外硬件逻辑；TVM 提供异构后端调优但需要付出更高调优成本。[R1]

Govorkova 等针对 Microchip PolarFire 辐射硬 FPGA 的 hls4ml 后端，展示了硬件感知量化和自动 HLS 转换的可行性；该论文针对轻量自编码器而非 VLM，但提供了“从算法到辐射硬 FPGA”闭环的工程先例。[R22]

### 4.3 必须避免的写法

- 不写“模型降低 x% FLOPs，因此星上时延降低 x%”；
- 不写“INT4 因此功耗必然降低到四分之一”；
- 不用服务器 GPU 的吞吐结果替代 CubeSat 或辐射硬平台结果；
- 不以单次冷启动 benchmark 代替热稳态和长时间任务负载。

### 4.4 最低报告指标

| 类别 | 最低报告项 | 目的 |
| --- | --- | --- |
| 编译 | 支持/回退算子比例、动态形状策略 | 揭示实际可执行性 |
| 内存 | 静态分配、峰值、碎片、加载时间 | 验证 DRAM 闭合 |
| 能耗 | 平均/峰值 W、J/image、J/token | 进入轨道预算 |
| 热控 | 温度、降频点、稳态吞吐 | 区分峰值和持续能力 |

## 5. 动态资源治理：从 early exit 到任务模式切换

### 5.1 early exit 的正确定位

BranchyNet 将中间分类器用于高置信样本提前退出，以减少推理时间和能耗。[R21a] LayerSkip 通过训练阶段的 layer dropout 和 early-exit loss，使 LLM 在早层退出或进行自推测解码；其论文在若干任务上报告最高 2.16× 加速。[R21]

对单星 VLM，early exit 不应只被表述为网络结构技巧，而应放入任务级调度：不同任务价值、资源状态与置信度对应不同执行深度。

### 5.2 推荐的四级运行模式

| 模式 | 典型任务 | 主要执行组件 |
| --- | --- | --- |
| M0 | 云/模糊/无效数据筛选 | 规则或轻量视觉模型 |
| M1 | 分类、检测、变化判断 | 视觉编码器 + 小任务头 |
| M2 | 结构化短报告 | 压缩 VLM + 受限输出 |
| M3 | 高价值解释或复杂问答 | 完整/较完整 VLM + KV 管理 |

### 5.3 星上调度变量

运行时至少应读取：可用电量、温度、可用 DRAM、任务截止期、链路状态、输入复杂度和模型置信度。基于这些变量，系统可改变分辨率、ROI 数量、token 预算、最大输出长度、模型层数和是否推迟/卸载任务。

SpaceVerse 和 Grace 虽不是纯单星部署论文，但提供了重要系统边界：两者均采用紧凑星上 LVLM 处理低成本任务、地面大模型处理复杂任务，并根据置信度与链路条件决定任务路径。SpaceVerse 报告相对比较基线的平均准确率提升与时延降低；Grace 在真实轨道数据驱动的实验中报告显著降低平均时延。[R3][R4]

**正文边界**：这两项工作宜被用来说明“星上小模型 + 地面大模型兜底”的任务分级思想，不应被写成完整 VLM 已在单星上常态化运行的证据。

## 6. 推理期可靠性与容错运行

### 6.1 故障对象需要分层

| 故障对象 | 直接风险 | 建议保护 |
| --- | --- | --- |
| 权重/LoRA 包 | 模型语义偏移、启动失败 | 哈希、签名、A/B 回滚 |
| 激活/中间矩阵 | 单次推理错误 | ABFT、范围检测、重算 |
| Softmax/GELU/LayerNorm | 非线性误差放大 | 区间/近似保护、保守精度 |
| KV cache | 长序列错误累积 | block 校验、关键 token 固化 |
| 编译产物/运行时状态 | 异常退出、错误算子 | watchdog、checkpoint、CPU 回退 |

### 6.2 已有算法证据

Xue 等的 ViT 软错误研究发现，ViT 在线性 GEMM/全连接部分具有相对均匀的脆弱性，但 Softmax 和 GELU 等非线性计算更脆弱；其提出的分块 ABFT 与范围保护方案可在小开销下提高错误环境中的准确率。[R23]

ReaLM 面向 LLM 推理，先以故障注入刻画模型容错性，再采用统计 ABFT 和低成本错误检测电路；论文报告 1.42% 面积与 1.79% 功耗开销下显著降低困惑度退化，并提升能效。[R24]

**星上适配判断**：上述研究不等于空间辐射验证，但为“只保护高敏感组件、用软件/算法减少冗余”的选择性容错策略提供直接技术依据。空间场景还必须叠加 TID/SEE、看门狗复位、存储器 ECC/EDAC、任务中断与不可维修约束。

### 6.3 单星最小可靠性闭环

```text
加载前：模型/Adapter 哈希与版本验证
运行中：温度、电量、内存、错误计数监控
异常时：checkpoint → 重试 → 轻量模型回退 → CPU/规则回退
恢复后：记录错误、保留任务证据、等待安全版本切换
```

### 6.4 正文可写结论

> 对单星 VLM 而言，可靠性不应只被理解为硬件防护。更可操作的方向是将错误敏感性分析、关键算子保护、模型完整性校验、检查点恢复和任务级降级共同纳入推理运行时，使系统在资源异常或软错误条件下优先维持可解释的最低任务能力。[R23][R24]

## 7. 与“持续适应：在轨微调”章节的接口

本章仅保留“模型如何安全加载、回滚和降级”的推理期内容；LoRA、Adapter、prompt、联邦聚合和漂移检测的训练策略应转入持续适应章节。

Pu 和 Xu 的 LoRA-Det 是两章之间的重要接口案例：其针对星载遥感目标检测，在仅更新 12.4% 参数时达到全量微调性能的 97%–100%，证明“上传小型参数增量而非全模型”具有明确的星地通信价值。[R25]

Du 等的在轨 GeoFM 研究也显示，跨传感器域差异可能显著损害任务表现，说明“压缩后能运行”与“在任务域中可靠”之间仍需域适配与持续更新机制。[R2]
